# 엔진 계산 전용 리팩토링 계획

## Context

현재 `MatchingEngine`은 `OrderBook`을 생성자로 받아 직접 mutate한다.
`EngineHandler`는 엔진 호출 → `SettlementService` DB 반영 → 캐시 갱신 순으로 처리하는데,
DB 실패 시에도 in-memory `OrderBook`은 이미 변경된 상태가 된다.

목표: 엔진이 "변경안을 계산"하고, 애플리케이션이 "DB 커밋 후 live 반영"하는 구조로 전환.
DB 실패 시 live `OrderBook`이 이전 상태를 유지하는 것이 핵심 불변식이다.

---

## 이슈별 결정 및 리스크

### 결정 1: 이름 충돌 — `OrderBookView` 신규 도입, 기존 `OrderBookSnapshot` 유지

`adapter/out/cache/OrderBookSnapshot`은 가격 레벨별 수량 집계용(캐시 표현)으로 그대로 유지.
계산 입력용 새 타입은 `OrderBookView`로 명명.

```java
// 신규: domain/model/
public record OrderBookView(Symbol symbol, List<Order> bids, List<Order> asks) {}
```

**리스크:**
- 리팩토링 계획 문서의 `OrderBookSnapshot` 명칭과 불일치
- 문서를 함께 업데이트하지 않으면 후속 개발자 혼란 가능
- **대응:** 본 작업 완료 후 계획 문서의 `OrderBookSnapshot` → `OrderBookView` 일괄 수정

---

### 결정 2: `PlaceCalculationResult` 신규 추가, `PlaceResult` 유지

`PlaceResult`는 `domain/service/dto`에 위치한 도메인 DTO로 그대로 유지.
`PlaceCalculationResult`는 `bookOperations`를 포함하는 애플리케이션 계층 DTO로 신규 추가.

```java
// 신규: application/engine/dto/
public record PlaceCalculationResult(
    List<Order> updatedOrders,
    List<Trade> trades,
    List<BookOperation> bookOperations
) {}
```

`EngineHandler`에서 `PlaceCalculationResult → SettlementInput` 변환 처리.
`SettlementService` 인터페이스는 변경 없음.

**리스크:**
- `updatedOrders`, `trades`가 `PlaceResult`와 `PlaceCalculationResult`에 중복
- 두 타입의 동기화 관리 부담 (필드 추가 시 양쪽 모두 수정)
- **대응:** `PlaceCalculationResult`를 애플리케이션 계층 전용으로 명확히 패키지 분리.
  장기적으로 `PlaceResult`를 내부 계산용으로 격하하고 `PlaceCalculationResult`를 유일 반환 타입으로 정리하는 Phase E에서 처리.

---

### 결정 3: `MatchingEngine`에서 `OrderBook` 필드 제거

생성자 파라미터에서 `OrderBook` 제거. 계산 메서드가 `OrderBookView`를 입력으로 받는 구조.

```java
// 변경 전
public final class MatchingEngine {
    private final OrderBook orderBook;
}

// 변경 후
public final class MatchingEngine {
    // 필드 없음 — 순수 계산 함수 집합
}
```

`EngineContext`에서 1줄 변경:
```java
// 변경 전
MatchingEngine matchingEngine = new MatchingEngine(orderBook);
EngineHandler engineHandler = new EngineHandler(symbol, matchingEngine, orderBook, ...);

// 변경 후
MatchingEngine matchingEngine = new MatchingEngine();
EngineHandler engineHandler = new EngineHandler(symbol, matchingEngine, orderBook, ...);
```

**리스크:**
- `MatchingEngine`이 stateless가 되면 심볼별 인스턴스가 불필요해짐
- Spring Bean으로 올릴 수 있으나, `EngineContext` 구조 재검토 범위가 커짐
- **대응:** 이번 리팩토링에서는 `EngineContext` 내 수동 생성 방식 유지.
  Spring Bean 전환은 별도 티켓으로 분리.

### 결정 4: `OrderBook.update(Order)` — 방식 B (live Order 필드 복사)

`OrderBook` 내 `Deque`와 `index`는 동일 참조를 공유한다.
`ArrayDeque`는 임의 위치 삽입이 불가능하므로 참조 교체는 O(n) 재구성이 필요.
대신 live Order의 필드를 복제본 값으로 덮어쓰는 방식을 선택.

`Order.applyFrom(Order calculated)` 추가:
```java
// Order.java에 추가
void applyFrom(Order calculated) {
    this.status = calculated.getStatus();
    this.remaining = calculated.getRemaining();
    this.cumBaseQty = calculated.getCumBaseQty();
    this.cumQuoteQty = calculated.getCumQuoteQty();
}
```

`OrderBook.updateInPlace(Order updatedOrder)` 추가:
```java
// index에서 live Order 참조를 꺼내 applyFrom 호출
// Deque 구조 불변 → FIFO 위치 완벽 유지
public void updateInPlace(Order updated) {
    Order live = index.get(updated.getOrderId());
    if (live != null) live.applyFrom(updated);
}
```

**리스크:**
- `Order`에 package-private `applyFrom()`이 추가되어 mutable surface가 넓어짐
- "계산 전용 엔진" 원칙과 약간 충돌: 엔진이 순수 계산을 해도 결국 live Order를 mutate함
- **대응:** `applyFrom()`을 package-private 또는 `OrderBook` 전용으로 제한.
  장기적으로 `Order`를 immutable record로 전환 시 이 부분을 재검토.

### 결정 5: `Order.copy()` — `restore()` 위임 없이 독립 구현

`Order` 필드는 전부 record/primitive value type이므로 shallow copy = deep copy.
`restore()`는 "DB 재구성"의 의미를 가지므로 `copy()`와 의도가 다름.

```java
// Order.java에 추가
public Order copy() {
    return new Order(
        orderId, accountId, clientOrderId, acceptedSeq,
        side, symbol, orderType, tif,
        price, quoteQty, quantity,
        remaining, status, cumQuoteQty, cumBaseQty,
        orderedAt, createdAt, updatedAt
    );
}
```

`OrderBookViewFactory`에서 스냅샷 생성 시 `order.copy()` 호출로 참조 분리.

**리스크:**
- `Order` 생성자에 필드가 추가될 경우 `copy()`도 수동 업데이트 필요
- `restore()`와 `copy()` 두 곳에서 동기화 누락 시 버그
- **대응:** `Order` 생성자는 private이므로 컴파일 오류로 즉시 감지 가능.
  추가 시 IDE refactoring으로 안전하게 처리.

---

### 결정 6: Phase B 과도기 전략 — 기존 메서드 유지 없이 한 Phase 교체

기존 `place*()` 메서드를 유지하면서 `calculatePlace()`를 병행하면
`MatchingEngine` 내에 `this.orderBook`(없어짐)과 `input.view()` 두 소스가 공존한다.
과도기 상태가 길어질수록 실수로 기존 메서드를 호출하는 버그 위험 증가.

전략:
1. Phase C 시작 전에 `EngineHandler` 통합 테스트를 먼저 작성 (안전망)
2. `MatchingEngine` 기존 메서드를 `calculate*` 메서드로 한 번에 교체
3. 기존 `PlaceResult` 반환 대신 `PlaceCalculationResult` 반환으로 전환

**리스크:**
- 한번에 많은 변경 → PR 규모가 커짐
- 통합 테스트 없는 상태에서 교체 시 회귀 감지 어려움
- **대응:** Phase C 진입 전 `EngineHandler` 통합 테스트가 필수 선행 조건.
  테스트 없으면 Phase C를 시작하지 않는다.

## Phase별 구현 순서

### Phase A: 준비 (기반 타입)

1. `OrderBookView` 도입 (`domain/model/`)
2. `BookOperation` 도입 (`application/engine/dto/`) — sealed interface: Add/Remove/Replace
3. `Order.copy()` 추가
4. `Order.applyFrom(Order)` 추가 (package-private)
5. `OrderBook.updateInPlace(Order)` 추가

### Phase B: 계산 분리 (MatchingEngine 교체)

6. `PlaceCalculationInput`, `CancelCalculationInput` 도입 (`domain/service/dto/`)
7. `PlaceCalculationResult`, `CancelCalculationResult` 도입 (`application/engine/dto/`)
8. `OrderBookViewFactory` 도입 — `order.copy()` 기반 참조 분리 스냅샷 생성
9. `MatchingEngine`에서 `OrderBook` 필드 제거
10. `MatchingEngine.calculatePlace(PlaceCalculationInput)` 구현 — 내부 `OrderBookView` 기반 계산
11. `MatchingEngine.calculateCancel(CancelCalculationInput)` 구현
12. `EngineContext`에서 `MatchingEngine` 생성자 변경

### Phase C: 반영 분리 (EngineHandler 재작성)

> **선행 조건:** `EngineHandler` 통합 테스트 작성 완료 후 진입

13. `EngineResultPersistenceService` 도입
    - `persistPlaceResult(PlaceCalculationResult)`: orders/trades/balances 트랜잭션
    - `persistCancelResult(CancelCalculationResult.Cancelled)`: order/balance 트랜잭션
    - lock ordering 유지: `SettlementResult.ofDeltaBook()` 정렬 순서 그대로 준수
14. `OrderBookProjectionApplier` 도입
    - `add` → `orderBook.add()`
    - `remove` → `orderBook.remove()`
    - `replace` → `orderBook.updateInPlace()`
15. `EngineHandler` 재작성 — 계산 → 영속 → live 반영 → 캐시 순서 엄수

### Phase D: 복구 강화

16. `OrderRepository.findOpenOrdersBySymbol(Symbol)` 추가
17. `OrderBook.from(List<Order>)` 정적 팩토리 추가
18. `OrderBookRebuilder` 도입
19. `OrderBookProjectionApplier.apply()` 실패 시 engine-thread 즉시 rebuild 경로
20. rebuild 실패 시 dirty 표시 + 운영 알람 경로

### Phase E: 정리

21. 구 `MatchingEngine` place/cancel mutable API 완전 제거 (Phase B에서 이미 제거됨이라면 skip)
22. `SettlementService` 역할 재검토 — `EngineResultPersistenceService`로 완전 이관 여부 결정
23. 리팩토링 계획 문서의 `OrderBookSnapshot` → `OrderBookView` 용어 일괄 수정
24. 테스트 및 티켓 갱신

## 예외 정책

| 실패 지점 | live OrderBook 상태 | 대응 |
|----------|-------------------|------|
| `calculatePlace/Cancel()` | 변경 없음 | 로그 후 해당 커맨드만 drop |
| `persistPlaceResult/CancelResult()` | 변경 없음 | 로그 후 캐시 갱신 생략 |
| `apply()` | DB 반영됨, live 불일치 | engine-thread에서 즉시 `OrderBookRebuilder.rebuild()` |
| rebuild 실패 | dirty | 심볼 dirty 표시 + 운영 알람 |

## 검증

### 단위 테스트
- `MatchingEngine.calculatePlace()` — snapshot 기반 입력으로 updatedOrders/trades/bookOperations 검증, live OrderBook mutation 없음 확인
- `OrderBookProjectionApplier` — add/remove/replace 각각 적용 후 OrderBook 상태 검증
- `Order.copy()` — 원본 mutate 후 복제본 불변 확인

### 통합 테스트 (`EngineHandler` 기준)
- DB 반영 성공 시만 live OrderBook 변경 및 캐시 갱신
- DB 반영 실패 시 live OrderBook 이전 상태 유지 확인
- DB 반영 실패 시 캐시 갱신 생략 확인
- cancel skip(이미 체결된 주문) 처리
- `apply()` 실패 → rebuild → 캐시 재동기화 확인

### 복구 테스트
- DB open orders 기반 `OrderBook.from()` rebuild 후 live OrderBook 일치
- replay 후 캐시 일치
