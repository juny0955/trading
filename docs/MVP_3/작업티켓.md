## 2. 상세 티켓

### MVP3-001 B-1 고아 주문 해소 (`submit` 선행)

- 설명: `placeOrder()`를 `submit()` 성공 후 `saveInMemory()`로 변경
- 완료조건:
  - `submit()` 실패 시 저장소 미기록
  - 고아 주문 재현 케이스 방지 테스트 통과
- 리스크: 참조 공유 경로 영향

### MVP3-002 B-2 입력 검증 400 표준화

- 설명: 잘못된 `side/orderType`, LIMIT `price=null`을 `ValidationException(400)`으로 변환
- 완료조건:
  - 케이스별 400 + 에러코드 검증
  - 500 누수 없음
- 리스크: 예외 핸들러 중복 매핑

### MVP3-003 B-3 취소 API 응답 202 정합화

- 설명: `DELETE /orders/{orderId}`를 비동기 계약에 맞게 `202 Accepted`로 변경
- 완료조건:
  - 컨트롤러/통합 테스트 통과
- 리스크: 클라이언트 계약 변경 영향

### MVP3-004 TIF 도메인 확장 (`GTC/IOC/FOK`)

- 설명: `TimeInForce`, `Order.tif`, 요청 검증 경로 추가
- 완료조건:
  - LIMIT은 `GTC/IOC/FOK` 허용
  - MARKET에 `tif` 명시 시 `PlaceOrderValidator` 검증 실패로 `INVALID_REQUEST(400)`
- 리스크: 기존 주문 생성 경로 영향

### MVP3-005 IOC 매칭 규칙 구현

- 설명: 즉시 체결 후 잔량 취소, 오더북 미적재
- 완료조건:
  - IOC 부분체결 후 `CANCELLED`
  - IOC 전량체결 `FILLED`
- 리스크: 상태 전이 누락

### MVP3-006 FOK 사전 충족성 검사 구현

- 설명: `totalAvailableQty()`로 충족 검사 후 매칭/취소 분기
- 완료조건:
  - 미충족 시 체결 0건 + `CANCELLED`
  - 충족 시 전량체결
- 리스크: 오더북 깊이 증가 시 선형 스캔 레이턴시

### MVP3-007 MARKET BUY `quantity/quoteQty` 매칭 구현

- 설명: MARKET BUY 입력을 `quantity`/`quoteQty` XOR로 검증하고 모드별 매칭/상태 규칙을 구현
- 완료조건:
  - `quoteQty` 모드: `remainingQuote`/`executedTradeCount` 기반 매칭
  - `quoteQty` 모드 상태 확정: `executedTradeCount > 0` -> `markFilledByMarketBuy()`, `= 0` -> `cancel()`
  - `quantity` 모드: 기존 MARKET 주문과 동일하게 `remaining` 기반 동작
  - MARKET BUY 입력 XOR 위반(동시 누락/동시 입력) 시 `PlaceOrderValidator` 검증 실패로 `INVALID_REQUEST(400)`
- 리스크: `remaining` 기반 기존 `fill()` 의미와 충돌

### MVP3-008 MARKET BUY 응답 필드 확장

- 설명: 비동기 계약 기준으로 POST/GET 응답 역할을 분리하고, 조회 응답에 체결 결과 필드를 노출
- 완료조건:
  - `POST /orders`: `202 Accepted` + `orderId` 접수 응답만 반환
  - `GET /orders/{orderId}`: `requestedQuoteQty`/`requestedQty`, `cumQuoteQty`, `cumBaseQty`, `leftoverQuoteQty` 노출
  - `leftoverQuoteQty = requestedQuoteQty - cumQuoteQty` 일관성 검증
  - MARKET BUY `remaining=0` 해석 충돌 없음(quoteQty 모드)
- 리스크: 직렬화/호환성 변경

### MVP3-009 백프레셔 예외 표준화

- 설명: 큐 포화 시 `EngineQueueFullException` + `ENGINE_BACKPRESSURE(503)` 매핑
- 완료조건:
  - 큐 포화 유도 테스트에서 503 일관 응답
- 리스크: 재시도 트래픽 급증

### MVP3-010 `clientOrderId` 멱등 처리 (Phase 3)

- 설명: in-memory에서 `clientOrderId` 단독 유일키로 중복 제출 방지
- 완료조건:
  - 동일 `clientOrderId` 재시도 시 기존 `orderId` 반환
  - 주문 중복 미발생
- 리스크: 재시작 시 맵 유실

### MVP3-011 회귀/통합 검증

- 설명: Phase 2 회귀 + MVP3 기능 통합 검증
- 완료조건:
  - `docs/MVP_3/목표.md` DoD 전 항목 충족
- 리스크: 병합 후 상태 전이 회귀
