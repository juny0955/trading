# 예외 처리 가이드

> 마지막 수정: 2026-03-04

---

## 1. 설계 원칙

- **HTTP 상태 코드는 `@RestControllerAdvice`에서만 결정**한다. 예외 클래스 자체에 HTTP 개념을 넣지 않는다.
- **도메인 예외는 Spring 프레임워크에 의존하지 않는다.** `DomainException` 계층은 순수 Java다.
- **`ErrorCode`는 애플리케이션 레이어 예외의 계약(contract)이다.** HTTP 상태, 에러 코드 문자열, 기본 메시지를 정의한다.
- 모든 API 응답에는 **traceId**가 포함되어 로그-응답을 연결할 수 있다.

---

## 2. 예외 계층 구조

```
RuntimeException
├── DomainException(errorCode: String, message: String)   ← HTTP 의존 없음
│   ├── ConflictException          → HTTP 409 Conflict
│   └── BusinessRuleException      → HTTP 422 Unprocessable Entity
│
└── BusinessException(errorCode: ErrorCode)               ← ErrorCode가 HTTP 보유
    ├── NotFoundException          → HTTP 404 Not Found
    └── 그 외 직접 사용            → ErrorCode.status() 기반
```

### 선택 기준

| 상황 | 사용 예외 | HTTP |
|------|----------|------|
| 도메인 상태 충돌 (이미 완료된 주문 취소) | `ConflictException` | 409 |
| 도메인 규칙 위반 (시장가 주문은 취소 불가) | `BusinessRuleException` | 422 |
| 리소스 없음 (주문 조회 실패) | `NotFoundException` | 404 |
| 필드 단위 입력 검증 실패 | `@Valid` → `MethodArgumentNotValidException` (자동) | 400 |
| 교차 필드 검증 실패 (필드 조합 규칙) | 커스텀 `ConstraintValidator` → `MethodArgumentNotValidException` (자동) | 400 |
| 애플리케이션 레이어 오류 (기타) | `BusinessException` | ErrorCode 기반 |

## 3. ErrorCode 인터페이스

```java
public interface ErrorCode {
    HttpStatus status();   // HTTP 상태 코드
    String code();         // 에러 코드 문자열 (예: "ORDER_NOT_FOUND")
    String message();      // 기본 메시지 (동적 정보 없는 경우)
}
```

도메인별로 enum을 구현한다.

```java
// order/exception/OrderErrorCode.java
public enum OrderErrorCode implements ErrorCode {
    ORDER_NOT_FOUND(HttpStatus.NOT_FOUND, "ORDER_NOT_FOUND", "Order not found"),
    UNSUPPORTED_SYMBOL(HttpStatus.BAD_REQUEST, "UNSUPPORTED_SYMBOL", "Unsupported symbol");
    ...
}
```

**네이밍 규칙:** `{DOMAIN}_{NOUN}_{CONDITION}` 형식을 권장한다.
예: `ORDER_NOT_FOUND`, `ORDER_CANCEL_FORBIDDEN`, `SYMBOL_NOT_SUPPORTED`

## 4. 예외 작성 방법

### 4-1. 도메인 예외 (HTTP 의존 없음)

```java
// 상태 충돌 → ConflictException
public class OrderAlreadyFinalizedException extends ConflictException {
    public OrderAlreadyFinalizedException(String orderId) {
        super("ORDER_ALREADY_FINALIZED", "Order is already finalized: " + orderId);
    }
}

// 규칙 위반 → BusinessRuleException
public class OrderNotCancellableException extends BusinessRuleException {
    public OrderNotCancellableException(String orderId) {
        super("ORDER_NOT_CANCELLABLE", "Order is not cancellable (MARKET order): " + orderId);
    }
}
```

### 4-2. 애플리케이션 예외 (ErrorCode 필요)

```java
// 리소스 없음 → NotFoundException
public class OrderNotFoundException extends NotFoundException {
    public OrderNotFoundException(String orderId) {
        super(OrderErrorCode.ORDER_NOT_FOUND, "Order not found: " + orderId);
    }
}

// 애플리케이션 레이어 400 → BusinessException (ErrorCode.status() = BAD_REQUEST)
public class UnsupportedSymbolException extends BusinessException {
    public UnsupportedSymbolException(String symbol) {
        super(OrderErrorCode.UNSUPPORTED_SYMBOL, "Unsupported symbol: " + symbol);
    }
}
```

### 4-3. 교차 필드 검증 (ConstraintValidator)

단일 필드로 표현할 수 없는 조합 규칙은 클래스 레벨 `ConstraintValidator`로 구현한다.
`@Valid` 경로를 타므로 `GlobalExceptionHandler` 수정 없이 자동으로 400으로 처리된다.

```java
// adapter/validation/annotation/ValidPlaceOrder.java
@Target(ElementType.TYPE)
@Retention(RetentionPolicy.RUNTIME)
@Constraint(validatedBy = PlaceOrderValidator.class)
public @interface ValidPlaceOrder {
    String message() default "Invalid order request";
    Class<?>[] groups() default {};
    Class<? extends Payload>[] payload() default {};
}

// adapter/validation/PlaceOrderValidator.java
public class PlaceOrderValidator implements ConstraintValidator<ValidPlaceOrder, PlaceOrderRequest> {
    @Override
    public boolean isValid(PlaceOrderRequest req, ConstraintValidatorContext ctx) {
        if (req.tif() != null && "MARKET".equals(req.orderType())) {
            ctx.disableDefaultConstraintViolation();
            ctx.buildConstraintViolationWithTemplate("TIF is not allowed for MARKET orders")
               .addPropertyNode("tif")
               .addConstraintViolation();
            return false;
        }
        return true;
    }
}
```

### 4-3. 새 도메인 추가 시 체크리스트

1. `XxxErrorCode` enum을 `{domain}/exception/` 아래에 생성 (`implements ErrorCode`)
2. 개별 예외 클래스를 적절한 부모에 상속
3. `GlobalExceptionHandler` 수정 불필요 — 자동으로 처리됨

## 5. 예외 throw 위치 규칙

| 레이어 | 허용 예외 | 예시 |
|--------|----------|------|
| Domain (Entity/VO) | `ConflictException`, `BusinessRuleException` | 주문 상태 전이 위반 |
| Application Service | `NotFoundException`, `BusinessException` | 주문 조회 실패, 심볼 미지원 |
| Adapter (Controller) | `@Valid` + `ConstraintValidator` → `MethodArgumentNotValidException` (자동) | 필드 검증, 교차 필드 조합 규칙 |

> **원칙:** 도메인 레이어에서 `NotFoundException`을 throw하지 않는다. 리소스 조회는 서비스 레이어의 책임이다.

## 6. 에러 응답 스키마

```json
{
  "errorCode": "ORDER_NOT_FOUND",
  "message": "Order not found: ORD-0001",
  "traceId": "a1b2c3d4e5f6g7h8",
  "timestamp": "2026-03-02T10:00:00Z",
  "fieldErrors": null
}
```

검증 실패 시 (`@Valid` 또는 `ConstraintValidator`):

```json
{
  "errorCode": "INVALID_REQUEST",
  "message": "Validation failed",
  "traceId": "a1b2c3d4e5f6g7h8",
  "timestamp": "2026-03-02T10:00:00Z",
  "fieldErrors": [
    { "field": "quantity", "message": "must be greater than 0", "rejectedValue": -1 }
  ]
}
```

**`traceId`** 는 응답 헤더 `X-Trace-Id`와 서버 로그에 동일하게 기록된다. 이슈 발생 시 이 값으로 로그를 조회한다.

## 7. GlobalExceptionHandler 핸들러 우선순위

Spring은 가장 구체적인(하위) 타입의 핸들러를 선택한다.

```
MethodArgumentNotValidException → handleMethodArgumentNotValid → 400
NotFoundException               → handleNotFound               → 404
BusinessException               → handleBusiness               → ErrorCode.status()
ConflictException               → handleConflict               → 409
BusinessRuleException           → handleBusinessRule           → 422
Exception                       → handleUnexpected             → 500
```

> `DomainException`에 대한 핸들러는 없다. `DomainException`을 직접 throw하지 말고 반드시 하위 타입(`ConflictException` 또는 `BusinessRuleException`)을 사용한다.

## 8. traceId 흐름

```
HTTP Request
    │
    ▼
TraceIdFilter          → UUID 생성 → MDC("traceId") 등록
                                    → 응답 헤더 X-Trace-Id 추가
    │
    ▼
Controller / Service / Domain
    │ (예외 발생)
    ▼
GlobalExceptionHandler → MDC.get("traceId") → ErrorResponse.traceId 포함
    │
    ▼
HTTP Response (traceId 포함)
```

로그 설정에서 `%X{traceId}`를 패턴에 추가하면 모든 로그에 traceId가 자동 포함된다.

```yaml
# application.yml
logging:
  pattern:
    console: "%d{HH:mm:ss} [%X{traceId}] %-5level %logger{36} - %msg%n"
```

## 9. 향후 확장 계획

| 항목 | 시점 | 내용 |
|------|------|------|
| `InfrastructureException` | 외부 시스템 연동 시 | `dependency`, `retryable` 필드 포함 |
| `ContextKey` enum | context 사용 케이스 발생 시 | PII 누출 방지용 화이트리스트 키 |
| Message 템플릿 | 다국어 지원 필요 시 | `errorCode` 기반 메시지 번들 |
| ErrorCode 번호 규칙 | 도메인 수 증가 시 | `ORDER_001_NOT_FOUND` 형식 |
