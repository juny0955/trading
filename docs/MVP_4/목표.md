# Trading Exchange Engine - MVP Phase 4 목표

## 0. 범위 고정

> Phase 4는 거래 엔진을 실제 거래 시스템 구조로 전환하는 단계이며, 핵심은 정합성 / 영속성 / 복구 / 멱등성이다.

- [ ] 주문 접수 경계와 엔진 결과 반영 경계를 분리한다
- [ ] in-memory 참조 공유 전제 없이도 동작하는 구조로 전환한다
- [ ] Account / Balance / Hold 도메인 도입
- [ ] 주문 소유(Account) 개념 도입
- [ ] 주문 접수 시 잔고 검증 및 hold reserve
- [ ] 체결 시 정산 처리 및 취소 시 hold 해제
- [ ] DB 기반 상태 영속성 전환
- [ ] `(accountId, clientOrderId)` 기준 멱등성 보장
- [ ] 서버 재시작 시 open order replay 복구
- [ ] Phase 4 TPS 기준선 측정 및 병목 확인

### 제외 범위

- [ ] Outbox 이벤트 파이프라인
- [ ] 운영 메트릭 / 모니터링 고도화
- [ ] FOK 성능 최적화
- [ ] 이벤트 기반 비동기 후처리
- [ ] 취소/거절 사유의 DB 감사 필드 고도화
- [ ] 동일 가격 레벨 내 live 순서 완전 재현

- [ ] `trades.symbol` 컬럼 포함 고정

  - `CANCELLED` 전환
  - hold 복구
  - idempotency key 삭제

## 2. 계정 / 잔고 / 홀드 도메인

### 2.1 주문 소유(Account)

- [ ] `PlaceOrderCommand.accountId` 추가
- [ ] 주문은 반드시 Account 소유로 생성
- [ ] 주문 조회 / 취소는 account scope에서 수행

### 2.2 Balance 모델

- [ ] `Balance { asset, available, held }` 도입
- [ ] `available + held` 보존 규칙 유지
- [ ] 자산별 잔고 조회 / 잠금 / 업데이트 경로 분리

### 2.3 Hold 정책

- [ ] `LIMIT BUY`
  - `price * quantity`를 quote asset에 hold
- [ ] `LIMIT SELL`
  - `quantity`를 base asset에 hold
- [ ] `MARKET BUY (quoteQty)`
  - `quoteQty`를 quote asset에 hold
- [ ] `MARKET SELL`
  - `quantity`를 base asset에 hold
- [ ] 주문 접수 단계에서는 주문 유형별로 필요한 자산 1개 row만 잠금
- [ ] 체결 정산 / 취소 반영 단계에서 복수 balance row lock 필요 시 정렬된 lock 순서 사용

## 3. 주문 접수 / 멱등성

### 3.1 주문 접수 트랜잭션

- [ ] orderId 선할당
- [ ] idempotency key insert (`orderId` 포함)
- [ ] account 존재 검증
- [ ] 잔고 검증
- [ ] hold reserve
- [ ] order insert (`ACCEPTED`)
- [ ] commit 후 engine queue submit

### 3.2 멱등성

- [ ] `clientOrderId`를 API / DB 모두 필수로 고정
- [ ] `(account_id, client_order_id)` unique 보장
- [ ] `idempotency_keys.client_order_id`: `NOT NULL`
- [ ] `idempotency_keys.order_id`: `NOT NULL`
- [ ] 조회 후 생성이 아니라 `insert-first` 기준으로 처리
- [ ] `idempotency key insert`는 hold reserve / order insert와 동일 트랜잭션에서 수행
- [ ] 중복 요청 시 기존 order 반환
- [ ] queue submit 실패 시 idempotency row까지 정리

### 3.3 주문 상태 모델

- [ ] `orders.status`
  - `ACCEPTED`
  - `NEW`
  - `PARTIALLY_FILLED`
  - `FILLED`
  - `CANCELLED`
- [ ] `REJECTED`는 `orders.status`가 아니라 DB row 미생성 요청 결과로 분리
- [ ] `CANCELLED`는 종료 사유를 세분화하지 않은 통합 종료 상태로 사용

## 5. 정산 / 취소 / 복구

### 5.1 정산 규칙

- [ ] trade delta 반영
  - BUY 체결: quote held 감소 / base available 증가
  - SELL 체결: base held 감소 / quote available 증가
- [ ] hold 반환 delta 반영
  - LIMIT BUY 가격 개선 차액 반환
  - MARKET BUY 미사용 예산 반환
  - 취소 시 잔여 hold 반환
- [ ] 원주문 hold 기준값은 `updatedOrders`의 불변 필드(`price × quantity`, `quoteQty` 등)로 정산 서비스 내부에서 재계산
- [ ] 실제 반환 금액은 `trades + updatedOrders` 최종 상태와 함께 계산 (`SettlementInput`에 별도 스냅샷 전달 없음)

### 5.2 취소

- [ ] `DELETE /accounts/{accountId}/orders/{orderId}`는 비동기 계약
- [ ] queue submit 성공 직후 `202 Accepted` 반환
- [ ] 최종 상태는 `GET /accounts/{accountId}/orders/{orderId}`로 확인
- [ ] 취소 완료 시 잔여 hold 해제

### 5.3 재시작 복구

- [ ] replay 대상은 `NEW`, `PARTIALLY_FILLED`
- [ ] `ACCEPTED`는 replay 대상에서 제외
- [ ] `accepted_seq` 기준으로 open order replay
- [ ] 오더북 집계 상태(depth/잔량)와 DB 주문 상태 복구
- [ ] orphan `ACCEPTED`는 `CANCELLED`로 정리
- [ ] orphan `ACCEPTED`의 hold 복구 및 idempotency key 삭제
- [ ] Phase 4에서는 orphan 정리 사유를 DB 컬럼이 아니라 애플리케이션 로그로만 남김

## 7. 테스트 목표

### 7.1 기능 / 정합성 테스트

- [ ] account 없는 주문 요청 거절

- [ ] `(accountId, clientOrderId)` 중복 요청 시 주문 중복 미발생
- [ ] queue submit 실패 시 order / hold / idempotency 롤백
- [ ] LIMIT BUY 가격 개선 차액 반환
- [ ] MARKET BUY `quoteQty` 미사용 예산 반환
- [ ] 부분체결 후 취소 시 잔여 hold 반환
- [ ] trade 영속 저장 및 조회 가능
- [ ] account 기준 trade 조회 시 `orders` join 경로 동작
- [ ] replay 후 open order 재구성
- [ ] orphan `ACCEPTED` 정리 및 hold 복구

### 7.2 불변식 검증

- [ ] 잔고 불변식
  - 각 `(account_id, asset)`에 대해 `available + held` 정합성 유지
- [ ] hold 불변식
  - 종료된 주문(`FILLED`, `CANCELLED`)의 잔여 hold = 0
- [ ] 상태 불변식
  - `FILLED` 주문은 `remaining = 0`
  - 체결 없는 취소(`cum_base_qty = 0`)는 `remaining = quantity`
  - 부분체결 후 취소는 `0 < remaining < quantity` 가능
- [ ] replay 불변식
  - 재시작 후 `ACCEPTED` 상태 주문 = 0
  - open order로부터 재계산한 held 합계 == `balances.held` 합계

## 8. 완료 기준 (DoD)

- [ ] 주문 생성/취소와 엔진 결과 반영 책임이 분리되어 있다
- [ ] 참조 공유 없이도 주문 상태 영속 반영의 의미가 일관된다
- [ ] 주문이 Account 소유로 생성되고 잔고 검증 후 접수된다
- [ ] 주문 접수 시 hold reserve가 수행되고 부족 잔고는 거절된다
- [ ] 체결 / 취소 / 종료에 따라 hold / available 정산이 일관되게 반영된다
- [ ] 주문 / 체결 / 잔고 상태가 DB에 영속된다
- [ ] `(accountId, clientOrderId)` 멱등성이 보장된다
- [ ] queue submit 실패 시 order / hold / idempotency가 롤백된다
- [ ] 서버 재시작 후 오더북의 집계 상태(depth/잔량), DB 주문 상태, 잔고/hold 상태가 일관되게 유지된다
- [ ] 동일 가격 레벨 내 live 순서 완전 재현은 Phase 4 완료 기준에 포함하지 않는다
- [ ] open order로부터 재계산한 held 합계와 `balances.held` 합계가 일치한다
- [ ] TPS 측정 계획의 동일 테스트는 최소 3회 반복 수행하고 중앙값 기준으로 병목 지점을 식별한다
- [ ] Phase 5 제외 범위 기능이 Phase 4 구현에 유입되지 않는다
