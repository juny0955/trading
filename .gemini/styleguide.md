# 코드 리뷰 가이드

## 1. 리뷰 핵심 원칙

* **객관적 시각**: 작성자의 의도를 존중하되, 코드 자체의 결함이나 개선점에 집중합니다.
* **근거와 제안**: 모든 피드백에는 명확한 **이유(Why)**와 구체적인 **개선안(How)**이 포함되어야 합니다.
* **한국어 사용**: 소통 효율을 위해 모든 리뷰와 코멘트는 한국어로 작성합니다.

---

## 2. 리뷰 작성 템플릿 (근거 및 제안)

리뷰 작성 시 아래의 구조를 권장합니다.

> **[태그] 요약**
> * **근거**: (왜 이 코드가 수정되어야 하는지, 어떤 문제가 예상되는지 기술)
> * **제안**: (어떻게 수정하면 좋을지 코드 스니펫이나 구체적인 방법 제시)
>
>

### 예시

* **[P1] 성능 저하 우려**
* **근거**: 현재 루프 내에서 매번 데이터베이스 조회가 발생하고 있어, 데이터 양이 많아질 경우 $O(n)$의 쿼리 부하가 예상됩니다.
* **제안**: 상단에서 `Map` 등을 이용해 데이터를 캐싱하거나, `JOIN` 쿼리를 사용하여 한 번에 조회하도록 변경해 주세요.

---

## 3. 리뷰 우선순위 태그

| 태그 | 의미 | 대응 방식 |
| --- | --- | --- |
| **[P1]** | **Must Fix** | 기능 오류, 보안 이슈 등 반드시 수정이 필요한 사항 |
| **[P2]** | **Should Fix** | 가독성, 유지보수성, 컨벤션 관련 개선 권고 사항 |
| **[P3]** | **Consider** | 단순 질문이나 미래를 위한 가벼운 아이디어 제안 |
