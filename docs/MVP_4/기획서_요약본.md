# MVP Phase 4 요약본

## 1. 목표

Phase 4의 목표는 매칭 엔진 프로토타입을 **운영 가능한 거래 시스템 구조**로 전환하는 것이다.

핵심 목표:

* `Account / Balance / Hold` 도메인 도입
* 주문/체결/잔고의 DB 영속성 도입
* `(accountId, clientOrderId)` 기준 멱등성 보장
* 엔진 결과를 정합하게 DB에 반영

시스템은 단일 정산 자산 마켓만 지원한다. 현재 정산 자산 코드는 `KRW`이며, 계정 잔고는 정산 자산과 거래 심볼 자산 단위로 관리한다. 복수 quote 자산 및 환전은 지원하지 않는다.

---

## 2. Phase 4 범위

포함:

* 계정 소유 주문 모델
* 잔고 검증
* hold reserve / release
* 체결 정산
* 주문/체결/잔고 DB 저장
* 재시작 시 replay 복구
* account 기준 멱등 처리

제외:

* Outbox
* 운영 메트릭/모니터링
* FOK 성능 최적화
* 이벤트 기반 후처리

---

## 3. 핵심 구조

요청 처리:

1. `OrderCommandService`가 `accountId`, `clientOrderId`를 받는다
2. 애플리케이션 레이어에서 `orderId`를 선할당한다
3. 주문 접수 트랜잭션에서
   - idempotency key insert (`orderId` 포함)
   - balance lock
   - hold reserve
   - order insert (`ACCEPTED`)
4. commit 후 engine queue submit
   - submit 실패 시 보상 트랜잭션으로 order / hold / idempotency 정리
5. engine이 `PlaceResult(updatedOrders, trades)` 생성
6. Persistence Layer가
   - orders update
   - trades insert
   - balances update
   를 같은 트랜잭션으로 반영

저장소 책임:

* `AccountRepository`: account 조회
* `BalanceRepository`: 정산 자산 및 거래 심볼 자산 balance 조회/락/hold/update
* `OrderRepository`: 주문 저장/조회
* `TradeRepository`: 체결 저장/조회

---

## 4. 상태 모델

`orders.status` 허용 값:

* `ACCEPTED`
* `NEW`
* `PARTIALLY_FILLED`
* `FILLED`
* `CANCELLED`

설명:

* `ACCEPTED`: DB 저장은 됐지만 아직 엔진 미처리
* `NEW`: 엔진이 수신했고 오더북에 살아 있음
* `PARTIALLY_FILLED`: 일부 체결, 잔량 존재
* `FILLED`: 전량 체결 완료
* `CANCELLED`: 사용자 취소 / IOC 잔량 종료 / FOK 미충족 / 시장가 종료 등 종료 상태 통합 표현

별도 요청 결과:

* `REJECTED`: 주문 생성 전 거절된 요청 결과
  - `orders.status` 값이 아님

---

## 5. 멱등성

기준:

* unique key: `(account_id, client_order_id)`

방식:

1. orderId 선할당
2. idempotency key insert
3. 충돌 시 기존 order 반환
4. 성공 시 주문 생성 계속 진행

queue submit 실패 시:

* order / hold / idempotency를 모두 제거
* 동일 키 재시도 시 새 주문으로 진행 가능
* `503 ENGINE_BACKPRESSURE` 반환

---

## 6. 정산 규칙

초기 hold:

* LIMIT BUY -> `price * quantity`
* LIMIT SELL -> `quantity`
* MARKET BUY -> `quoteQty`
* MARKET SELL -> `quantity`

BUY hold는 정산 자산에 적용하고, SELL hold는 주문 심볼 자산에 적용한다. 현재 정산 자산 코드는 `KRW`다.
`MARKET BUY`는 `quoteQty`만 지원하며, `quantity` 기반 시장가 매수는 지원하지 않는다.

정산 원칙:

* 체결 delta는 `trades`로 계산
* hold 반환 delta는 `updatedOrders` 최종 상태로 계산
* 둘은 같은 트랜잭션에서 함께 반영

핵심 포인트:

* LIMIT BUY 가격 개선 차액 반환
* MARKET BUY `quoteQty` 미사용 예산 반환
* 부분체결 후 `CANCELLED` 가능

---

## 7. replay / 복구

재시작 시:

* `NEW`, `PARTIALLY_FILLED` 주문만 replay 대상
* `accepted_seq`를 replay용 안정 정렬 키로 사용
* 동일 가격 레벨 내 live 순서 완전 재현은 Phase 4 범위 밖
* 일치 기준은 `오더북 집계 상태(depth/잔량) + 주문 상태`

orphan `ACCEPTED` 처리:

* replay 대상 제외
* `CANCELLED` 전환
* hold 복구
* idempotency key 삭제
* Phase 4에서는 별도 감사 컬럼 없이 로그로만 남김

---

## 8. API 정책

주문 생성:

* `POST /accounts/{accountId}/orders`
* 비동기 계약
* queue submit 성공 직후 `202 Accepted` 반환

주문 취소:

* `DELETE /accounts/{accountId}/orders/{orderId}`
* 비동기 계약
* `202 Accepted` 반환

주문 조회:

* `GET /accounts/{accountId}/orders/{orderId}`
* 비동기 처리의 최종 상태 확인 용도

---

## 9. 완료 기준

* 주문이 account 소유로 생성된다
* 잔고 검증과 hold reserve가 적용된다
* 체결/취소 시 정산이 수행된다
* 주문 / 체결 / 잔고 상태가 DB에 저장된다
* `(accountId, clientOrderId)` 멱등성이 보장된다
* queue submit 실패 시 주문 / hold / idempotency가 모두 정리된다
* 재시작 후 replay로 오더북이 복구된다
* orphan `ACCEPTED` 주문이 `CANCELLED`로 정리된다
* 오더북 집계 상태, DB 주문 상태, 잔고/hold 상태가 일관된다
* open order로부터 재계산한 held 합계와 `balances.held` 합계가 정합하다
