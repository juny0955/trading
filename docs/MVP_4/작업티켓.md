## 2. 상세 티켓

### MVP4-001 Account / Balance / Hold 도메인 도입

- 설명: Account 소유 개념과 Balance/Hold 모델을 도입하고, 주문 유형별 hold 정책을 도메인 규칙으로 고정한다.
- 완료조건:
  - `Account`, `Balance` 모델 정의 완료
  - `available`, `held` 규칙 정리 완료
  - `LIMIT BUY`, `LIMIT SELL`, `MARKET BUY(quoteQty)`, `MARKET SELL` hold 정책 반영
  - `MARKET BUY(quantity)` 비지원 정책 문서/코드 반영
  - BUY hold는 정산 자산, SELL hold는 주문 심볼 자산에 적용된다는 규칙 반영
  - 주문 접수 단계는 단일 asset row lock, 정산 단계는 필요 시 정산 자산 + 주문 심볼 자산 복수 row lock이라는 규칙 문서/코드 반영
- 리스크:
  - 기존 주문 도메인과 계정 도메인 경계가 모호해질 수 있음

### MVP4-002 주문 접수 / 취소 책임 분리

- 설명: 현재 `OrderCommandService`에 함께 몰린 주문 생성/취소 책임을 Phase 4 경계에 맞게 분리한다.
- 완료조건:
  - `PlaceOrderUseCase`와 `CancelOrderUseCase`가 별도 서비스 경계로 분리된다
  - 주문 접수 로직에 account 검증 / hold reserve / idempotency / submit rollback이 추가되어도 단일 클래스가 과도하게 비대해지지 않는다
  - 취소 로직이 주문 접수 로직의 부속 경로가 아니라 독립 유스케이스로 유지된다
  - 엔진 디스패치 책임과 영속 반영 책임의 경계가 코드에서 드러난다
- 리스크:
  - 책임 경계가 불명확하면 Phase 4 변경이 하나의 서비스에 누적됨

### MVP4-003 account scope API / Command 모델 정리

- 설명: account 기반 주문 생성/조회/취소 계약으로 API와 Command 모델을 정리한다.
- 완료조건:
  - `PlaceOrderCommand.accountId` 추가
  - `POST /accounts/{accountId}/orders`
  - `DELETE /accounts/{accountId}/orders/{orderId}`
  - `GET /accounts/{accountId}/orders/{orderId}` 계약 반영
  - `clientOrderId` 필수화
- 리스크:
  - 기존 API 경로와 응답 계약 변경 영향

### MVP4-004 DB 스키마 초안 및 Repository 영속 계층 전환

- 설명: 인메모리 저장소를 DB 스키마/Repository 구조로 전환한다.
- 완료조건:
  - `orders`, `trades`, `balances`, `accounts`, `idempotency_keys` 스키마 정의
  - `orders`에 `account_id`, `order_type`, `tif`, `quote_qty`, `cum_base_qty`, `cum_quote_qty`, `client_order_id`, `accepted_seq` 반영
  - `accepted_seq`는 DB 자동 생성이 아니라 application이 acceptance 시점에 채워 저장
  - `trades`에 `symbol`, `price`, `quantity`, `buy_order_id`, `sell_order_id` 반영
  - `idempotency_keys`에 `account_id`, `client_order_id`, `order_id`, `created_at` 반영
  - `orders.client_order_id`: `NOT NULL`
  - `idempotency_keys.client_order_id`: `NOT NULL`
  - `idempotency_keys.order_id`: `NOT NULL`
  - `OrderRepository`, `TradeRepository`, `AccountRepository`, `BalanceRepository` DB 구현 골격 마련
  - Repository 계약이 메모리 참조 공유가 아니라 명시적 저장/반영 의미를 갖도록 정리
- 리스크:
  - 기존 참조 공유 전제와 DB 영속 semantics가 충돌할 수 있음

### MVP4-005 주문 접수 트랜잭션 구현 (검증 / hold / order / 멱등)

- 설명: 주문 접수 시 `orderId`를 선할당하고, `idempotency key insert -> 잔고 검증 -> hold reserve -> order insert`를 하나의 트랜잭션으로 고정한다.
- 완료조건:
  - 애플리케이션 레이어에서 `orderId` 선할당
  - 애플리케이션 레이어에서 `acceptedSeq` 할당
  - `idempotency key insert(orderId 포함)`가 트랜잭션 초반에 수행
  - 주문 접수 트랜잭션 구현
  - `ACCEPTED` 상태 order 저장
  - balance row `SELECT FOR UPDATE` 적용
  - hold reserve 실패 시 전체 롤백
  - 트랜잭션 롤백 시 idempotency key도 함께 롤백
- 리스크:
  - lock wait 증가 및 account 단위 병목

### MVP4-006 queue submit 실패 보상 롤백 구현

- 설명: DB commit 후 engine queue submit 실패 시 order / hold / idempotency를 보상 롤백하고 `503`을 반환한다.
- 완료조건:
  - queue submit 실패 시 order row 삭제
  - hold rollback
  - idempotency key 삭제
  - `queue_full_rollback_count` 측정 가능 상태 확보
- 리스크:
  - 보상 실패 시 orphan 상태 발생

### MVP4-007 멱등성 경쟁 / 중복 반환 / 계측 보강

- 설명: MVP4-004에서 구현된 insert-first 멱등 경로 위에 경쟁 상황 처리, 기존 주문 반환, 계측을 보강한다.
- 완료조건:
  - `(account_id, client_order_id)` unique 보장
  - 중복 요청 시 기존 order 반환
  - idempotency conflict를 측정 가능하도록 계측 포인트 추가
  - queue submit 성공 후 응답 실패 재시도 시에도 기존 order 반환 보장
- 리스크:
  - 동시 요청 시 unique 충돌 처리 미흡

### MVP4-008 엔진 결과 영속 반영 및 SettlementService 구현

- 설명: 엔진이 반환한 `updatedOrders`, `trades`를 기준으로 정산 계산 후 DB에 반영한다.
- 완료조건:
  - `SettlementService`에서 trade delta + hold 반환 delta 계산
  - 엔진 디스패치와 엔진 결과 영속 반영 책임이 분리된다
  - KRW + 주문 심볼 자산 balance row lock 순서(asset 이름 정렬) 적용
  - engine result transaction 구현
  - `orders update`, `trades insert`, `balances update` 수행
  - LIMIT BUY 가격 개선 차액 반환, MARKET BUY 미사용 예산 반환 반영
- 리스크:
  - partial fill / cancel / leftover 처리 누락 시 잔고 드리프트

### MVP4-009 account 기준 Trade 조회 경로 구현

- 설명: MVP4-008에서 insert된 trade를 기반으로 account 기준 조회 경로를 구현한다. trade 영속 insert 자체는 MVP4-008 책임이다.
- 완료조건:
  - order 기준 trade 조회 가능
  - account 기준 trade 조회 시 `trades -> orders` join 경로 구현
  - trade 스키마 정규화 유지 (account 정보는 join으로 조회, 중복 저장 없음)
- 리스크:
  - 조회 성능 저하 가능성

### MVP4-010 취소 비동기 계약 및 hold 해제 구현

- 설명: 취소를 비동기 계약으로 정리하고, 취소 완료 시 잔여 hold를 해제한다.
- 완료조건:
  - `DELETE /accounts/{accountId}/orders/{orderId}`가 `202 Accepted` 반환
  - 취소 요청 queue submit 후 비동기 처리
  - `GET /accounts/{accountId}/orders/{orderId}`로 최종 상태 확인 가능
  - 취소 완료 시 hold 해제
  - 이미 종료된 주문 취소 시 reject 처리
- 리스크:
  - HTTP 응답과 실제 상태 반영 사이의 해석 차이

### MVP4-011 replay 복구 및 orphan ACCEPTED 정리 구현

- 설명: 재기동 시 open order를 replay하고 orphan `ACCEPTED`를 정리한다.
- 완료조건:
  - startup recovery / replay 책임이 `EngineManager` 생명주기 로직에 과도하게 누적되지 않는다
  - replay 대상: `NEW`, `PARTIALLY_FILLED`
  - `ACCEPTED`는 replay 대상에서 제외
  - `accepted_seq` 기준 replay
  - 오더북 집계 상태(depth/잔량) 복구
  - orphan `ACCEPTED -> CANCELLED`
  - orphan hold 복구 및 idempotency key 삭제
  - orphan 정리는 애플리케이션 로그로 기록
- 리스크:
  - replay 순서 근사화로 동일 가격 레벨 내부 순서 차이 발생

### MVP4-012 정합성 / 불변식 / 회귀 테스트

- 설명: Phase 4 핵심 정합성과 회귀를 테스트로 고정한다.
- 완료조건:
  - 잔고 불변식 검증
  - hold 불변식 검증
  - 부분체결 후 취소 불변식 검증
  - replay 후 `ACCEPTED = 0` 검증
  - open order로부터 재계산한 held 합계 == `balances.held`
  - Phase 3 회귀 테스트 유지
- 리스크:
  - 시나리오 누락 시 운영 버그가 테스트를 통과할 수 있음

### MVP4-013 TPS 측정 포인트 추가 및 기준선 측정

- 설명: Phase 4 TPS 계획에 맞는 계측 포인트를 추가하고 기준선 측정을 수행한다.
- 완료조건:
  - `place_order_tps`, `cancel_order_tps`, `accepted_order_tps`, `cancel_completed_tps`, `cancel_rejected_tps` 측정 가능
  - `queue_wait_latency`, `engine_processing_latency`, `order_accept_tx_latency`, `engine_result_tx_latency`, `end_to_end_order_latency`, `end_to_end_cancel_latency` 측정 가능
  - `engine_queue_depth`, `engine_backpressure_count` 측정 가능
  - `db_lock_wait_time`, `balance_lock_contention_count`, `db_deadlock_count`, `db_commit_failure_count`, `db_rollback_count`, `idempotency_conflict_count`, `queue_full_rollback_count`, `error_rate` 측정 가능

  - 최소 3회 반복 수행 후 중앙값 기준으로 결과 기록
- 리스크:
  - 로그 기반 계측만으로는 후처리 비용 증가
