# MVP Phase 4 구현 전 아키텍처 검토

## 1. 목적

이 문서는 `docs/MVP_4/기획서.md`가 아니라 **현재 구현된 코드베이스**를 기준으로,
Phase 4 착수 전에 구조적으로 먼저 인지해야 할 개선 포인트를 정리한 문서다.

목표는 다음 두 가지다.

1. Phase 4 요구사항(Account / Balance / Hold / DB / Replay / Idempotency)이 들어올 때
   어디서 구조적 충돌이 발생할지 미리 식별한다.
2. 구현 중 대규모 재작업이 발생하지 않도록, 변경 우선순위를 사전에 정리한다.

---

## 2. 현재 구조 요약

현재 주문 생성 흐름은 대략 다음과 같다.

1. `OrderController`가 `/orders` API로 요청을 받는다.
2. `OrderCommandService`가 메모리 멱등성 검사, `Order.create()`, `EngineManager.submit()`, `OrderRepository.save()`를 수행한다.
3. `EngineManager`가 심볼별 `EngineContext`를 찾아 큐에 커맨드를 넣는다.
4. `EngineLoop`가 단일 스레드에서 커맨드를 꺼내 `EngineHandler`로 전달한다.
5. `EngineHandler`가 `MatchingEngine`을 호출하고 결과를 `OrderRepository`에 반영한다.

이 구조는 Phase 3 요구사항에는 적합하지만, 다음 전제에 크게 의존한다.

* 메모리 기반 멱등성
* in-memory repository의 참조 공유
* 엔진 처리와 영속 반영의 결합
* account scope 부재
* replay / startup recovery 부재

---

## 3. 개선 포인트

### 3.1 `OrderCommandService`의 책임이 이미 과도하게 집중되어 있다

현재 `OrderCommandService`는 다음을 동시에 수행한다.

* 메모리 멱등성 관리
* 주문 생성
* 엔진 submit
* 최초 저장
* 취소 검증 및 취소 submit

근거:

* `clientOrderMap`으로 메모리 멱등성을 직접 관리한다.
  * `src/main/java/dev/junyoung/trading/order/application/service/OrderCommandService.java:26-27`
* `placeOrder()` 안에서 `Order.create()`, `engineManager.submit()`, `orderRepository.save()`를 모두 수행한다.
  * `src/main/java/dev/junyoung/trading/order/application/service/OrderCommandService.java:53-68`
* `cancelOrder()`도 같은 클래스에서 주문 조회, 취소 가능성 검증, 엔진 submit까지 처리한다.
  * `src/main/java/dev/junyoung/trading/order/application/service/OrderCommandService.java:78-89`

왜 문제인가:

* Phase 4에서는 여기에 account 검증, balance lock, hold reserve, DB idempotency, queue submit 실패 보상까지 추가된다.
* 주문 생성과 취소는 실패 조건과 트랜잭션 경계가 다르므로, 한 서비스에 같이 두면 변경 영향 범위가 커진다.

권장 방향:

* `OrderCommandService`를 유지하더라도 내부 책임은 최소한 아래 수준으로 분리하는 것이 좋다.
  * 주문 접수 유스케이스
  * 취소 유스케이스
  * 멱등성 처리
  * queue submit 실패 보상

---

### 3.2 엔진 처리와 영속 반영이 `EngineHandler`에 결합되어 있다

현재 `EngineHandler`는 엔진 디스패치만 하지 않는다.
매칭 결과를 바로 `OrderRepository`에 저장하고, 취소도 상태 변경 후 바로 저장한다.

근거:

* `PlaceResult result = processPlaceOrder(order);`
  * `src/main/java/dev/junyoung/trading/order/application/engine/EngineHandler.java:53-55`
* `result.updatedOrders().forEach(orderRepository::save);`
  * `src/main/java/dev/junyoung/trading/order/application/engine/EngineHandler.java:55`
* 취소도 `engine.cancelOrder()` 후 즉시 `orderRepository.save(cancelled)`를 호출한다.
  * `src/main/java/dev/junyoung/trading/order/application/engine/EngineHandler.java:59-62`

왜 문제인가:

* Phase 4에서는 엔진 결과 반영이 단순 `save(order)`가 아니라
  `orders update + trades insert + balances update + 정산 계산`이 된다.
* 이 책임이 계속 `EngineHandler`에 남아 있으면, 엔진 디스패처가 트랜잭션 오케스트레이터로 비대해진다.

권장 방향:

* `EngineHandler`는 가능한 한 `MatchingEngine -> PlaceResult / CancelResult` 생성까지만 담당한다.
* 결과 반영은 별도 애플리케이션 서비스로 분리하는 편이 좋다.

---

### 3.3 현재 repository 계약은 “영속 계층”보다 “객체 보관소”에 가깝다

현재 `MemoryOrderRepository`는 `ConcurrentHashMap`에 동일 객체 참조를 보관한다.
그리고 `OrderCommandService` 주석과 `EngineHandler` 주석 모두 이 참조 공유에 의존하고 있다.

근거:

* `MemoryOrderRepository.save()`는 `orders.put(...)`만 수행한다.
  * `src/main/java/dev/junyoung/trading/order/adapter/out/persistence/MemoryOrderRepository.java:13-18`
* `OrderCommandService`는 “참조 공유로 이후 상태 변경 자동 반영”이라고 명시한다.
  * `src/main/java/dev/junyoung/trading/order/application/service/OrderCommandService.java:63-65`
* `EngineHandler`도 “참조 공유로 OrderRepository에 자동 반영된다”고 설명한다.
  * `src/main/java/dev/junyoung/trading/order/application/engine/EngineHandler.java:45-47`

왜 문제인가:

* DB 영속성으로 전환하면 같은 객체를 공유해서 상태가 자동 반영되는 모델은 더 이상 성립하지 않는다.
* 지금 계약을 유지한 채 DB adapter를 붙이면, 저장 시점과 상태 변경 시점의 의미가 불명확해진다.

권장 방향:

* Phase 4에 들어가기 전, `save()`를 “현재 상태를 명시적으로 반영하는 동작”으로 보는 방향으로 사고를 바꿔야 한다.
* 엔진 내부 객체 참조와 영속 상태 반영은 분리된 경계라는 점을 코드 구조에도 드러내는 것이 좋다.

---

### 3.4 엔진 시작 경로가 replay / 복구 확장을 받기 어려운 형태다

현재 `EngineManager`는 `@PostConstruct`에서 바로 `EngineContext`를 만들고 엔진을 시작한다.
`EngineContext`는 생성자 내부에서 queue, orderBook, matchingEngine, handler를 모두 조립한다.

근거:

* `EngineManager.start()`가 `new EngineContext(...)` 후 즉시 `ctx.start()`를 호출한다.
  * `src/main/java/dev/junyoung/trading/order/application/engine/EngineManager.java:42-50`
* `EngineContext` 생성자에서 `OrderBook`, `MatchingEngine`, `EngineHandler`, `EngineLoop`를 한 번에 조립한다.
  * `src/main/java/dev/junyoung/trading/order/application/engine/EngineContext.java:29-36`

왜 문제인가:

* Phase 4의 replay 복구는 “DB 조회 -> orphan ACCEPTED 정리 -> open order 정렬 -> OrderBook 주입 -> 엔진 시작” 순서를 요구한다.
* 지금 구조에서는 부팅 훅과 엔진 조립 로직이 강하게 붙어 있어, 복구 절차를 넣는 순간 `EngineManager`가 비대해질 가능성이 높다.

권장 방향:

* startup recovery 책임을 `EngineManager` 밖으로 분리하는 것이 좋다.
* `EngineContext`도 “빈 오더북 생성”만 가능한 컨테이너가 아니라,
  외부에서 준비한 open order 상태로 초기화할 수 있는 구조를 염두에 둘 필요가 있다.

---

### 3.5 API / UseCase 경계가 `orderId-only` 중심이라 account scope 확장 비용이 크다

현재 REST API와 커맨드 모델에는 account 정보가 없다.

근거:

* `OrderController`는 `/orders`, `/orders/{orderId}` 경로만 사용한다.
  * `src/main/java/dev/junyoung/trading/order/adapter/in/rest/OrderController.java:22-23`
  * `src/main/java/dev/junyoung/trading/order/adapter/in/rest/OrderController.java:31-50`
* `cancelOrder()`와 `getOrder()`도 `orderId`만 받는다.
  * `src/main/java/dev/junyoung/trading/order/adapter/in/rest/OrderController.java:40-53`
* `PlaceOrderCommand`에는 `accountId`가 없다.
  * `src/main/java/dev/junyoung/trading/order/application/port/in/command/PlaceOrderCommand.java:11-20`

왜 문제인가:

* Phase 4에서는 주문 생성, 조회, 취소 모두 account scope 기준으로 바뀐다.
* 지금처럼 `orderId`만으로 주문을 찾는 패턴이 널리 퍼지면, 이후 account ownership 검증을 도입할 때 수정 범위가 커진다.

권장 방향:

* controller, use case, query service 모두에서 account scope를 먼저-class concern으로 올려야 한다.
* 특히 조회/취소 유스케이스는 `orderId-only`를 기본 전제로 더 퍼뜨리지 않는 것이 좋다.

---

### 3.6 조회용 파생 계산이 서비스 내부에 고정되어 있어 query 확장에 취약하다

현재 `OrderQueryService`는 조회뿐 아니라 응답용 파생 필드를 내부에서 직접 계산한다.

근거:

* `getOrder()`가 바로 `toResult(order)`를 호출한다.
  * `src/main/java/dev/junyoung/trading/order/application/service/OrderQueryService.java:17-23`
* `toResult()`, `deriveFields()`, `deriveQuoteModeFields()`, `deriveQuantityModeFields()`가 서비스 내부 private 메서드다.
  * `src/main/java/dev/junyoung/trading/order/application/service/OrderQueryService.java:29-72`

왜 문제인가:

* 지금은 주문 단독 조회만 있어서 괜찮지만, Phase 4에서는 account, trades, balances와 결합된 조회 요구가 늘어난다.
* 조회용 계산이 서비스 안에 깊게 박혀 있으면, query model 변화가 서비스 책임 확장으로 이어진다.

권장 방향:

* 응답 조립과 파생 필드 계산은 mapper/assembler 성격으로 분리할 여지가 있다.
* 특히 quoteQty leftover, cumulative fields 계산은 이후 정산/조회 규칙과 맞물릴 수 있어 분리 이점이 크다.

---

### 3.7 엔진 큐 경계로 도메인 엔티티 전체를 넘기는 방식은 Phase 4에서 무거워질 수 있다

현재 `EngineCommand.PlaceOrder`는 `Order` 엔티티 전체를 큐로 전달한다.

근거:

* `record PlaceOrder(Order order) implements EngineCommand`
  * `src/main/java/dev/junyoung/trading/order/application/engine/EngineCommand.java:23-29`
* `OrderCommandService`도 `new EngineCommand.PlaceOrder(order)`를 바로 생성해 submit한다.
  * `src/main/java/dev/junyoung/trading/order/application/service/OrderCommandService.java:63`

왜 문제인가:

* Phase 4에서 `Order`에 account ownership, clientOrderId, acceptedSeq, cumulative fields, 영속 관련 메타데이터가 더 붙는다.
* 엔진에 꼭 필요하지 않은 데이터까지 큐 경계로 넘기기 시작하면, 엔진과 주문 영속 모델의 결합이 심해진다.

권장 방향:

* 단기적으로는 유지해도 되지만, 최소한 엔진이 실제로 어떤 필드만 필요로 하는지 분리해서 보는 것이 좋다.
* Phase 4 구현 중 엔진 전용 command payload가 필요해질 가능성을 열어 두는 편이 안전하다.

## 4. 우선순위 제안

Phase 4 착수 직전에 반드시 큰 리팩터링을 먼저 해야 한다는 뜻은 아니다.
다만 아래 순서로 생각하고 들어가면 재작업을 줄일 수 있다.

### P0

* `OrderCommandService`의 책임 분리
* `EngineHandler`에서 영속 반영 책임 분리
* repository를 참조 공유 모델에서 끊어내는 방향 정리

### P1

* account scope 기준으로 controller / use case / command 시그니처 재정의
* startup recovery / replay를 위한 엔진 초기화 경계 분리

### P2

* query 조립 로직 분리
* 엔진 큐 payload 경량화 검토

---

## 5. 결론

현재 코드베이스는 Phase 3 목표에는 잘 맞는다.
다만 구조의 중심 전제가 아직도 아래에 묶여 있다.

* 메모리 멱등성
* 객체 참조 공유
* 엔진 처리와 영속 반영의 결합
* account scope 부재

따라서 Phase 4의 핵심 리스크는 “기능 추가” 자체보다,
이 전제를 유지한 채 DB / Account / Hold / Replay를 얹으려 할 때 발생하는 구조적 충돌이다.

착수 전 관점에서 가장 중요한 메시지는 다음 한 줄이다.

> Phase 4는 엔진 기능 확장이 아니라, 주문 접수 경계와 엔진 결과 반영 경계를 다시 세우는 작업이다.
