# MVP Phase 4 기획서

---

# 1. 목표

MVP Phase 3까지의 매칭 엔진은 다음 상태이다.

* 다종목 매칭 엔진 동작
* LIMIT / MARKET 주문 지원
* IOC / FOK 주문 규칙
* 엔진 단일 스레드 이벤트 루프
* 인메모리 상태 관리
* 테스트 기반 검증 완료

그러나 현재 시스템은 **엔진 수준의 프로토타입**이며 다음이 없다.

* 계정 및 잔고 관리
* 주문에 대한 자산 홀드
* 체결에 따른 자산 정산
* DB 기반 상태 영속성
* 서버 재시작 시 상태 복구

따라서 **MVP Phase 4의 목표는 엔진을 실제 거래 시스템 구조로 전환하는 것이다.**

핵심 목표는 다음 4가지이다.

1. **Account / Balance / Hold 도메인 도입**
2. **DB 기반 상태 영속성 전환**
3. **Account 기준 멱등성 보장**
4. **엔진 결과를 DB 트랜잭션으로 반영**

---

# 2. Phase 4 범위

Phase 4는 다음 영역만 포함한다.

* 계정 / 잔고 / 홀드 모델
* 주문 소유(Account) 개념 도입
* 주문 접수 시 잔고 검증
* 주문에 대한 자산 홀드
* 체결 시 정산 처리
* 주문 취소 시 홀드 해제
* DB 기반 영속성
* Account 기준 멱등 처리

시스템은 단일 정산 자산 마켓만 지원한다.
현재 정산 자산 코드는 `KRW`이며, 계정 잔고는 정산 자산과 거래 심볼 자산 단위로 관리한다. 복수 quote 자산 및 환전은 지원하지 않는다.

다음 항목은 **Phase 5에서 처리한다.**

* Outbox 이벤트 파이프라인
* 운영 메트릭 / 모니터링
* FOK 성능 최적화
* 이벤트 기반 비동기 후처리

# 2.1 Phase 4 선결정 항목

구현 착수 전 아래 항목은 먼저 고정한다.

* 재시작 시 open order replay 방식
* `(account_id, client_order_id)` 충돌 처리 방식
* 주문 접수 시 hold reserve 트랜잭션 경계
* LIMIT BUY 가격 개선 시 잔여 hold 반환 규칙
* MARKET BUY `quoteQty` 미사용 예산 hold 해제 규칙
* accountId HTTP 수신 방식: Path Variable (`/accounts/{accountId}/orders`)

* 오더북 재시작 복구 방식: OrderBook 직접 주입 (EngineContext 생성자)
* orders 테이블 `client_order_id` 컬럼: NOT NULL (API 및 DB 모두 필수)
* trades 테이블 `symbol` 컬럼: 추가
* Replay 안정 정렬 키: `accepted_seq` 추가

* 도메인 책임

* 영속 책임
  - `AccountRepository`, `BalanceRepository`
* 정산 계산 책임

핵심 변경점

* 주문은 **Account 소유**
* 체결 결과는 **DB에 영속**
* 잔고와 홀드는 **Account 도메인에서 관리**

## 주문 상태 정의

Phase 4에서 `orders.status` 컬럼은 다음 값만 가진다.

* `ACCEPTED`
  - 주문 접수 트랜잭션이 커밋되어 DB에 저장되었으나, 아직 엔진이 처리하지 않은 상태
  - hold reserve와 idempotency key 기록은 완료된 상태
* `NEW`
  - 엔진이 주문을 수신했고, 체결 없이 오더북에 살아 있는 상태
* `PARTIALLY_FILLED`
  - 일부 체결되었고 잔량이 오더북에 남아 있는 상태
* `FILLED`
  - 전량 체결 완료 상태
* `CANCELLED`
  - 취소 또는 주문 규칙(IOC/FOK/시장가 종료)에 의해 종료된 상태
  - Phase 4에서는 종료 사유를 세분화하지 않고, 비체결/잔량 반환 포함 종료 상태를 `CANCELLED`로 통합 표현한다

상태 전이 원칙:

* 일반 경로: `ACCEPTED -> NEW | PARTIALLY_FILLED | FILLED | CANCELLED`
* replay 대상은 `NEW`, `PARTIALLY_FILLED`만 포함한다

## 요청 처리 결과

`REJECTED`는 order status 값이 아니라 **주문 생성 전 거절된 요청 결과**로 취급한다.

* `orders.status` 컬럼에 저장되지 않는다
* DB 주문 레코드 자체가 생성되지 않는다
* 예: 요청 검증 실패, 잔고 부족, 주문 접수 단계 거절
* 세부 종료/거절 사유 추적은 Phase 5 운영 추적성 범위에서 확장한다

```
MemoryOrderRepository
```

Phase 4에서는 다음 테이블이 추가된다.

---

## Order

```
orders
 ├─ order_id
 ├─ account_id
 ├─ client_order_id  -- NOT NULL
 ├─ symbol
 ├─ side
 ├─ order_type
 ├─ tif
 ├─ price
 ├─ quote_qty
 ├─ quantity
 ├─ remaining
 ├─ cum_quote_qty
 ├─ cum_base_qty
 ├─ status
 ├─ accepted_seq     -- 전역 단조 증가 replay 정렬 키, 주문 acceptance 시점에 부여
 └─ created_at
```

필드 의도:

* `order_type`, `tif`
  - LIMIT / MARKET 및 GTC / IOC / FOK 복원용
* `quote_qty`
  - MARKET BUY 예산 모드 복원용
* `client_order_id`
  - API 및 DB 모두 필수값이다 (NOT NULL)
  - 멱등성 키 `(account_id, client_order_id)` unique constraint의 구성 요소이므로 null을 허용하지 않는다
* `cum_quote_qty`, `cum_base_qty`
  - 누적 체결 기준 정산 및 조회 응답 계산용

`accepted_seq` 정의:

* 주문이 `ACCEPTED`로 수용되는 시점에 application이 할당하는 전역 단조 증가 값이다
* symbol별로 분리하지 않는 전역 시퀀스다
* 매칭 우선순위 결정에는 사용하지 않는다. replay 시 동일 가격 레벨 내 삽입 순서 복원에만 사용한다
* `created_at`은 감사/조회용, `accepted_seq`는 replay 정렬 기준으로 역할을 분리한다
* Persistence layer는 할당된 `accepted_seq`를 그대로 저장한다. DB는 생성 주체가 아니라 영속화 주체다
* 단일 프로세스에서는 application 내부 시퀀스 발급기로 할당할 수 있다
* 멀티 인스턴스 환경에서는 DB sequence 같은 중앙 발급원을 application이 호출해 값을 받아 설정한다
* Phase 4에서는 `COMMIT -> queue submit -> engine 처리` 구조를 유지하므로,
  replay 시 `accepted_seq`를 live book entry 순서의 근사값으로 사용하는 단순화를 채택한다
* 따라서 replay 일치 기준은 **오더북 집계 상태와 주문 상태의 일치**이며,
  동일 가격 레벨 내 순서까지 완전 동일함을 Phase 4 DoD로 강제하지 않는다

## Trade

```
trades
 ├─ trade_id
 ├─ symbol
 ├─ price
 ├─ quantity
 ├─ buy_order_id
 ├─ sell_order_id
 └─ created_at
```

Trade account 저장 정책:

* Phase 4는 **정규화 유지**를 기본 원칙으로 한다

* Phase 4의 우선순위는 정합성 확보이며, 조회 최적화는 다음 단계로 이관한다

조회 정책:

* account 기준 trade 조회가 필요하면 `trades -> orders` 조인으로 처리한다
* Phase 4는 정합성 우선이며, 조회 최적화는 Phase 5 이후 검토한다

## Balance 동시성 보호

동일 account에서 동시 주문 접수 시 available 초과 사용을 방지하기 위해
비관적 락(`SELECT FOR UPDATE`)을 사용한다.

주문 접수 단계의 hold reserve에서는 주문 유형에 따라 필요한 자산 1개 행을 잠근다.

예:

* LIMIT BUY / MARKET BUY -> 정산 자산 행 lock (현재 코드는 `KRW`)
* LIMIT SELL / MARKET SELL -> 주문 심볼 자산 행 lock

동일 `accountId + asset` 기준 직렬화가 보장된다.

**KRW + 심볼 자산 lock 순서**:

체결 정산 또는 취소 반영 단계처럼 하나의 처리에서 복수 balance 행을 잠가야 하는 경우,
asset 이름 알파벳 오름차순으로 lock을 획득한다.

예: `BTC-KRW` 쌍 — BTC → KRW 순서로 잠근다.

이 순서를 모든 트랜잭션에서 동일하게 지키면 교차 lock으로 인한 데드락이 발생하지 않는다.

# 8. 엔진 결과 반영 방식

엔진은 **DB를 직접 수정하지 않는다.**

엔진은 다음 결과만 반환한다.

```
PlaceResult (엔진 반환)
 ├─ updatedOrders
 └─ trades
```

엔진은 잔고 도메인을 모른다. PlaceResult에는 `updatedOrders`와 `trades`만 포함한다.

Persistence Layer는 `trades`와 `updatedOrders`를 함께 사용해 잔고 delta를 계산한다.

BUY 체결 시:
- 매수자: KRW held 감소, 주문 심볼 자산 available 증가
- 매도자: 주문 심볼 자산 held 감소, KRW available 증가

SELL 체결 시:
- 매도자: 주문 심볼 자산 held 감소, KRW available 증가
- 매수자: KRW held 감소, 주문 심볼 자산 available 증가

계산 기준: `executionPrice × executedQty`

---

Persistence 계층이 다음을 수행한다.

```
DB Transaction
 ├─ orders update
 ├─ trades insert
 └─ balances update
```

이 트랜잭션은 **단일 트랜잭션**으로 수행된다.
`idempotency key insert`는 주문 접수 단계의 별도 트랜잭션에서 이미 완료되어 있어,
엔진 결과 반영 트랜잭션에는 포함되지 않는다.

---

## 잔고 반영 원칙

Persistence Layer는 두 종류의 효과를 함께 반영해야 한다.

1. `trades`로부터 계산되는 체결 기반 delta
2. `updatedOrders`의 최종 상태로부터 계산되는 hold 반환 delta

따라서 잔고 반영은 `trades`만 보고 계산하지 않는다.

예:

* IOC 잔량 취소 -> 잔량 hold 반환
* FOK 미충족 취소 -> 전체 hold 반환
* 외부 취소 -> 잔여 hold 반환
* MARKET BUY `quoteQty` 종료 -> 미사용 예산 반환
* LIMIT BUY 종료 -> 미체결 잔량 + 가격 개선 차액 반환

구현 원칙:

* 체결 delta와 hold 반환 delta는 같은 DB 트랜잭션에서 함께 반영한다
* `Trade`가 없는 상태 전이도 잔고 변화가 발생할 수 있음을 명시적으로 처리한다

---

## 재시작 복구

DB 영속성만으로는 재시작 이후 매칭을 이어갈 수 없다.
애플리케이션 부팅 시 미종결 주문을 읽어 심볼별 오더북을 재구성해야 한다.

복구 절차:

```
1. DB에서 status in (NEW, PARTIALLY_FILLED) 주문 조회
2. symbol별로 그룹화
3. replay 안정 정렬 키 + 우선순위 규칙에 따라 정렬
4. 각 symbol의 OrderBook 재구성
5. EngineContext / Cache 초기화
```

원칙:

* `FILLED`, `CANCELLED` 주문은 replay 대상이 아니다
* replay 결과는 DB 상태와 동일해야 한다
* 재기동 직후 조회 API와 실제 오더북 상태가 불일치하면 안 된다

**ACCEPTED 상태 잔존 주문 처리**:

재시작 시 status = ACCEPTED 주문은 엔진 진입이 보장되지 않은 orphan 주문으로 간주한다.

처리 정책:

* replay 대상에서 제외한다 (엔진이 수용한 적 없으므로 오더북 복원 대상 아님)
* CANCELLED로 일괄 전환한다
* held 감소, available 복구
* idempotency key 삭제 (동일 키 재시도 시 새 주문으로 진행 가능)
* orphan 정리 이력은 Phase 4에서는 애플리케이션 로그로만 남긴다
  - DB 스키마에 별도 취소 사유 컬럼은 추가하지 않는다

이 처리는 replay 시작 전 단계에서 수행한다.

## Replay 구현 방식

`EngineManager.start()`(`@PostConstruct`)에서 수행한다.

처리 순서:

```
1. DB에서 status IN (NEW, PARTIALLY_FILLED) 주문 조회
2. symbol별로 그룹화
3. `accepted_seq ASC`로 안정 정렬
4. EngineContext 생성 시 open order 목록을 생성자 파라미터로 전달
5. EngineContext 생성자 내부에서 OrderBook에 직접 주입
```

**정렬 기준 설명**:

Price-Time Priority는 두 레벨로 구성된다.

- BID: price DESC → time ASC (같은 가격에서 먼저 온 주문 우선)
- ASK: price ASC → time ASC

Replay 시 open order는 symbol별로 조회한 뒤 `accepted_seq ASC` 순으로 OrderBook에 삽입한다.
가격 우선순위는 OrderBook 내부 자료구조(TreeMap 등)가 유지하며,
동일 가격 레벨 내 시간 우선순위는 삽입 순서(`accepted_seq`)로 복원한다.
`created_at`은 감사/조회용 시각으로만 사용한다.

주의:

* `accepted_seq`는 live 시점의 엄밀한 book entry 순서가 아니라 replay용 안정 정렬 키다
* Phase 4에서는 price level / 잔량 / 상태 기준 정합성을 우선하고,
  동일 가격 레벨 내 완전한 순서 재현은 Phase 5 이후 개선 여지로 둔다

제약:

* Replay는 엔진 큐를 통하지 않는다
  (hold 검증 중복 실행 방지, 체결 없이 오더북만 복원)
* Replay 완료 전 엔진 스레드를 시작하지 않는다
* EngineContext 시그니처 변경:
  `EngineContext(Symbol, List<Order> openOrders, OrderRepository, OrderBookCache)`

## 주문 접수와 hold reserve

잔고 검증만 먼저 수행하고 hold reserve를 엔진 처리 이후로 미루면,
동일 account에서 동시 주문 시 초과 사용이 발생할 수 있다.

Phase 4에서는 주문 접수 단계에서 다음 순서로 처리한다.

```
1. orderId 선할당 (애플리케이션 레이어, TX 시작 전)
BEGIN TX
2. idempotency key insert 시도 (orderId 포함, 충돌 시 기존 order 반환)
3. account/balance 조회 (SELECT FOR UPDATE)
4. available 검증
5. hold reserve
6. `accepted_seq` 할당
7. order insert (ACCEPTED, orderId, acceptedSeq)
COMMIT
8. engine queue submit
```

제약:

* idempotency key를 트랜잭션 초반에 insert하여 동시 요청을 fail-fast로 차단한다
* `accepted_seq`는 주문이 영속화되기 전에 application이 명시적으로 채운다
* 가장 자연스러운 할당 지점은 `hold reserve` 직후, `order insert` 직전이다
* 이 위치는 주문 접수 성공 경계 안에 있으면서도 DB insert 시점의 부수효과에 의존하지 않는다
* `hold reserve`가 실패하면 트랜잭션 전체가 롤백된다
* queue submit 실패 시 hold / order / idempotency를 모두 제거해 동일 멱등키 재시도가 새 주문으로 진행되게 한다
* 동일 account의 balance row는 비관적 락(`SELECT FOR UPDATE`)으로 보호한다

## Queue submit 실패 시 처리

주문 접수 트랜잭션과 queue submit은 분리된다.

처리 순서:

```
1. orderId 선할당 (애플리케이션 레이어, TX 시작 전)
BEGIN TX
2. idempotency key insert 시도 (orderId 포함, 충돌 시 기존 order 반환)
3. account/balance 조회 (SELECT FOR UPDATE)
4. available 검증
5. hold reserve
6. `accepted_seq` 할당
7. order insert (ACCEPTED, orderId, acceptedSeq)
COMMIT

8. Queue submit
   try {
       engineManager.submit(symbol, PlaceOrder(order));
   } catch (EngineQueueFullException e) {
       compensationService.rollbackAcceptedOrder(order.orderId());
       throw e;
   }
```

보상 트랜잭션 (`compensationService.rollbackAcceptedOrder`):
- ACCEPTED order 삭제
- held 감소, available 복구
- idempotency key 삭제

정책:

* queue submit 실패는 영구 비즈니스 결과가 아니라 일시적 백프레셔로 간주한다
* 따라서 동일 `(account_id, client_order_id)` 재시도는 새 주문으로 진행될 수 있어야 한다
* 클라이언트에는 `503 ENGINE_BACKPRESSURE`를 반환한다
* queue submit 실패한 주문은 외부적으로 존재하지 않은 요청으로 간주한다. order 조회 대상에 남지 않는다. 실패 이력은 애플리케이션 로그에만 남긴다.

**삭제 vs CANCELLED 전환 정책 근거**:

* **queue submit 실패 시 → 주문/hold/idempotency 전체 삭제 (롤백)**
  - DB commit이 성공했더라도 엔진에 진입하지 못했으므로 외부적으로는 존재하지 않는 요청이다
  - 삭제 후 동일 `(account_id, client_order_id)`로 재시도 시 새 주문으로 처리할 수 있어야 한다

* **재시작 시 orphan ACCEPTED → CANCELLED 상태 전환**
  - 이미 commit된 이력이므로 삭제하면 상태 복구/정합성 판단 근거가 사라진다
  - 동일 `client_order_id`로 재시도를 허용하기 위해 idempotency key는 별도로 삭제한다
  - 취소 사유 같은 감사 필드는 Phase 5 운영 추적성 범위로 이관한다

실제 queue full 발생률은 매우 낮다(용량 10,000).
이 경로는 예외 처리 수준으로 구현한다.

**Submit 성공 후 응답 실패 시 재시도 보호**:

queue submit이 성공했으나 네트워크 장애 등으로 클라이언트가 응답을 받지 못한 경우,
클라이언트가 동일 `(account_id, client_order_id)`로 재시도하면
idempotency key 충돌로 기존 order가 반환된다. 중복 주문이 생성되지 않는다.

## 취소 요청 처리 흐름

`DELETE /accounts/{accountId}/orders/{orderId}` 요청의 처리 흐름이다.
주문 접수와 동일하게 **비동기 계약**을 사용한다.
즉, HTTP 응답은 queue submit 성공 직후 반환되며 엔진 처리와 DB 반영은 이후 비동기로 진행된다.

```
1. HTTP 요청 수신
2. account 소유권 검증 (orderId가 해당 accountId 소유인지 확인)
3. 엔진 큐에 CancelOrder 커맨드 submit
4. HTTP `202 Accepted` 응답 반환
5. 엔진이 주문을 CANCELLED 처리 후 CancelResult 반환
6. Persistence Layer가 DB에 반영
```

`202 Accepted` 의미:

취소 요청은 비동기 계약이므로 `202 Accepted`는 취소 완료가 아니라 취소 처리 시작을 의미한다.
클라이언트는 `202 Accepted` 수신 후 `GET /accounts/{accountId}/orders/{orderId}` 조회 API로 최종 상태(`CANCELLED` 여부)를 확인한다.
최종 상태 확인은 동일 account scope의 주문 조회 API(`GET /accounts/{accountId}/orders/{orderId}`)를 사용한다.
엔진 처리 또는 DB 반영 실패 시 주문 상태는 변경되지 않는다.

측정 기준:

* `http_cancel_order_latency`
  - HTTP 요청 수신 ~ queue submit 성공 후 `202 Accepted` 반환까지의 시간
* 취소의 실제 완료 시간은 별도 비동기 처리 지표로 본다
  - 엔진 처리 + DB 반영 완료 시점 기준

엔진 결과 반영:

```
CancelResult
 └─ cancelledOrder  (status = CANCELLED)
```

Persistence Layer는 `cancelledOrder`를 보고 hold를 반환한다.

```
held 감소
available 증가
order status → CANCELLED
```

이미 FILLED / CANCELLED 된 주문에 취소 요청이 오는 경우:

* 엔진이 해당 orderId를 오더북에서 찾지 못하면 오류를 반환한다
* Persistence Layer는 DB에서 현재 상태를 조회해 적절한 응답을 반환한다
  - FILLED: `400 ORDER_ALREADY_FILLED`
  - CANCELLED: `400 ORDER_ALREADY_CANCELLED`

역할 경계:

* 취소 가능 여부의 **1차 기준은 엔진 오더북 상태**다
* 엔진이 orderId를 오더북에서 찾지 못하면 취소를 거부한다
* **DB 조회는 보강 용도**다 — 이미 종료된 주문(FILLED, CANCELLED)에 대해 사용자에게 명확한 오류 메시지를 반환하기 위해서만 사용한다
* DB를 선행 기준으로 삼아서는 안 된다. 엔진 처리와 DB 반영 사이 비동기 구간에서 불일치가 발생할 수 있다

## 멱등성 경쟁 처리

권장 방식:

```
1. idempotency key insert 시도
2. unique key 충돌 시 기존 order_id 조회 후 반환
3. insert 성공 시에만 신규 주문 생성 계속 진행
```

또는 DB가 지원하면 `upsert`를 사용한다.

**트랜잭션 경계 원칙**:

idempotency key insert는 hold reserve, order 생성과 동일한 트랜잭션 내에서 수행한다.
insert 시 사용할 `order_id`는 주문 생성 전 애플리케이션에서 선할당한다.

```
BEGIN TX
 1. orderId 선할당 (애플리케이션)
 2. idempotency key insert (orderId 포함, unique key 충돌 시 조회 후 반환)
 3. account/balance 조회 (SELECT FOR UPDATE)
 4. available 검증
 5. hold reserve
 6. `accepted_seq` 할당
 7. order insert (ACCEPTED, orderId, acceptedSeq)
COMMIT
 8. engine queue submit
```

트랜잭션 롤백 시 idempotency key도 함께 롤백된다. dangling key가 발생하지 않는다.

원칙:

* 멱등성 보장은 애플리케이션 메모리가 아니라 DB unique key가 최종 기준이다
* 동일 `(account_id, client_order_id)` 요청은 어떤 경쟁 상황에서도 하나의 order만 생성해야 한다

## 부분체결 / 가격개선 정산 규칙

단순히 `hold 감소`, `balance 업데이트`만으로는 잔고 정합성이 부족하다.
주문 종료 상태별 반환 규칙을 명시한다.

### LIMIT BUY

초기 hold:

```
limitPrice * quantity
```

체결 시:

* 실제 체결가는 maker 가격을 사용한다
* `executedQuote = executionPrice * executedQty`
* hold에서 `executedQuote`만 차감한다

주문 종료 시:

* 미체결 잔량이 취소되면 남은 hold를 `available`로 반환한다
* 가격 개선(`executionPrice < limitPrice`)으로 남은 차액도 종료 시 반환한다
* GTC로 계속 열려 있는 주문은 잔여 hold를 유지한다

### LIMIT SELL

초기 hold:

```
quantity(base)
```

체결 시:

* hold에서 체결 수량만큼 차감한다
* KRW `available`을 증가시킨다

주문 종료 시:

* 미체결 잔량 취소분을 `available`로 반환한다
* GTC로 계속 열려 있는 주문은 잔여 hold를 유지한다

### MARKET BUY (`quoteQty`)

초기 hold:

```
quoteQty
```

`quantity` 기반 `MARKET BUY`는 지원하지 않는다.

체결 시:

* 실제 사용한 `executedQuote`만 hold에서 차감한다
* 체결된 base를 증가시킨다

주문 종료 시:

* `requestedQuoteQty - cumQuoteQty` 전액을 `available`로 반환한다

### MARKET SELL

초기 hold:

```
quantity(base)
```

체결 시:

* 체결 수량만큼 hold 차감
* 체결 대금만큼 quote 증가

주문 종료 시:

* 미체결 수량이 있으면 잔여 hold 반환

## 정산 구현 규칙

잔고 정산 서비스는 아래 입력을 함께 사용한다.

```
SettlementInput
 ├─ updatedOrders
 └─ trades
```

의도:

* `trades`로 체결 delta 계산
* `updatedOrders` 최종 상태로 hold 유지/해제 결정

**원주문 hold 기준값(originalHold) 계산**:

`originalHold`는 `SettlementInput` 전달값이 아니라 정산 서비스 내부에서 `updatedOrders`의 불변 필드로 재계산한다.

```
LIMIT BUY  → price × quantity
LIMIT SELL → quantity
MARKET BUY → quoteQty
MARKET SELL → quantity
```

주문 생성 이후 이 필드들은 변경되지 않으므로 정산 시점에 재계산해도 안전하다.
별도 스냅샷 전달이 불필요하며, `SettlementInput` 구조가 단순해진다.

## cum_quote_qty / cum_base_qty 갱신 시점

엔진은 주문 처리 중 누적 체결 수량을 내부적으로 추적하며,
`updatedOrders`에 포함된 `Order` 객체에 최신 `cumBaseQty`, `cumQuoteQty` 값을 담아 반환한다.

Persistence Layer는 `engine_result_tx` 내에서 이 값을 DB의 `cum_base_qty`, `cum_quote_qty` 컬럼에 기록한다.

따라서 `engine_result_tx_latency`에는 이 갱신 비용이 포함된다.

이 값들은 다음 용도로 사용된다:

* 정산 시 `requestedQuoteQty - cumQuoteQty` 계산 (MARKET BUY 미사용 예산 반환)
* 조회 API의 누적 체결 응답 필드 (`filledQty`, `filledQuoteQty`)

# 9. Phase 4 완료 기준

다음 조건이 만족되면 Phase 4가 완료된다.

* 주문이 Account 소유로 생성된다
* 주문 접수 시 잔고 검증이 수행된다
* 주문 생성 시 자산 hold가 적용된다
* 체결 시 잔고 정산이 수행된다
* 취소 시 hold가 해제된다
* 주문 / 체결 / 잔고 상태가 DB에 저장된다
* 서버 재시작 후 상태가 유지된다
* `(accountId, clientOrderId)` 멱등성이 보장된다
* 재시작 후 open order replay로 오더북이 복구된다
* 부분체결 / 가격개선 / 취소 시 hold 반환 규칙이 일관되게 적용된다
* accountId 기준 주문 접수 / 취소 API가 동작한다
* queue submit 실패 시 주문 / hold / idempotency가 모두 정리되어 동일 요청 재시도가 가능하다
* 서버 재시작 후 오더북의 집계 상태(depth/잔량), DB 주문 상태, 잔고/hold 상태가 일관되게 유지된다
* 재시작 복구 후 open order로부터 재계산한 held 합계와 `balances.held` 합계가 정합해야 한다
* 동일 가격 레벨 내 live 순서의 완전 재현은 Phase 4 완료 기준에 포함하지 않는다
* 재시작 시 orphan ACCEPTED 주문이 CANCELLED로 전환되고 hold / idempotency 상태가 일관되게 정리된다

# 11. Phase 4 구현 리스크

Phase 4 구현 중 주의해야 할 리스크이다. Phase 3의 문제점(메모리 의존, account 부재 등)은 Phase 4에서 해결 대상이므로 여기서는 다루지 않는다.

* **잔고 정합성**: 체결 delta와 hold 반환 delta가 동일 트랜잭션에서 처리되지 않으면 잔고 불일치 발생
* **replay 순서 오염**: `accepted_seq`가 실제 book entry 순서와 어긋나면 재시작 후 오더북 우선순위가 변경될 수 있음 (Phase 4에서는 근사값 사용으로 허용)
* **SELECT FOR UPDATE 데드락**: 동일 account의 KRW와 주문 심볼 자산 동시 lock 순서가 일치하지 않으면 데드락 발생 가능. lock 순서를 고정해야 한다
* **queue submit 롤백 타이밍**: commit 직후 ~ submit 전 프로세스 종료 시 ACCEPTED 상태 order가 DB에 잔존할 수 있음. 재시작 시 ACCEPTED 상태 처리 정책이 필요하다

Phase 4는 위 리스크를 인지하고 구현하여 **운영 가능한 거래 시스템 기반을 구축한다.**
