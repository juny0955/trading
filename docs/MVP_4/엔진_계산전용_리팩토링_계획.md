# 엔진 계산 전용 리팩토링 계획 (보완: Order 불변화 + 다중 인스턴스 대응)

## Context

현재 `MatchingEngine`은 `OrderBook`을 생성자로 받아 직접 mutate한다.
`EngineHandler`는 엔진 호출 → `SettlementService` DB 반영 → 캐시 갱신 순으로 처리하는데,
DB 실패 시에도 in-memory `OrderBook`은 이미 변경된 상태가 된다.

목표: 엔진이 "변경안을 계산"하고, 애플리케이션이 "DB 커밋 후 live 반영"하는 구조로 전환.
DB 실패 시 live `OrderBook`이 이전 상태를 유지하는 것이 핵심 불변식이다.

**보완 결정 2건이 기존 계획에 연쇄 영향을 준다:**

1. **Order → Immutable**: 상태전이 메서드(`fill`, `cancel`, `activate`)가 새 인스턴스를 반환해야 함
2. **다중 인스턴스 대응**: 나중에 확장할 수 있도록 아키텍처 경계를 설계
3. **Rust 재작성 대비**: `MatchingEngine` 및 계산 계층은 Spring 비의존 POJO로 유지해야 함

### 결정 2: `PlaceCalculationResult` 신규 추가, `PlaceResult` 유지

`PlaceResult`는 `domain/service/dto`에 위치한 도메인 DTO로 그대로 유지.
`PlaceCalculationResult`는 `bookOps`를 포함하는 애플리케이션 계층 DTO로 신규 추가한다.
비즈니스 거절은 예외가 아니라 결과 타입으로 표현한다.

`acceptedSeq`의 의미는 place 전용 주문 번호가 아니라,
**해당 symbol engine이 수용한 command sequence**로 통일한다.
따라서 place / cancel 모두 동일한 의미의 `acceptedSeq`를 가진다.

```java
// 신규: application/engine/dto/
public sealed interface PlaceCalculationResult
    permits PlaceCalculationResult.Accepted, PlaceCalculationResult.Rejected {

    record Accepted(
        Symbol symbol,
        long acceptedSeq,
        List<Order> updatedOrders,
        List<Trade> trades,
        List<BookOperation> bookOps
    ) implements PlaceCalculationResult {}

    record Rejected(
        Symbol symbol,
        long acceptedSeq,
        PlaceRejectCode reasonCode
    ) implements PlaceCalculationResult {}
}
```

`Accepted.updatedOrders` 범위:

- taker를 포함한다
- 상태가 변경된 모든 maker를 포함한다
- fully filled 되어 book에서 제거된 maker도 persistence 대상이므로 포함한다
- 상태 변화가 없는 resting order는 포함하지 않는다

`Accepted.updatedOrders` 순서 규약:

- 첫 원소는 항상 taker
- 이후에는 상태가 변경된 maker를 **실제 매칭 발생 순서**대로 나열한다
- replay / persistence / Rust 비교는 이 순서를 기준으로 한다

`Accepted.trades` 순서 규약:

- 매칭 루프에서 실제로 발생한 체결 생성 순서를 그대로 유지한다
- 저장 순서와 replay 순서도 이를 따른다

**리스크:**
- `updatedOrders`, `trades`가 `PlaceResult`와 중복
- **대응:** `PlaceCalculationResult`를 애플리케이션 계층 전용으로 명확히 패키지 분리.
  장기적으로 `PlaceResult`를 격하하고 `PlaceCalculationResult`를 유일 반환 타입으로 정리는 Phase E에서.
- `acceptedSeq` / `symbol` 같은 메타가 없으면 replay/debug 시 추적이 어려움
- **대응:** 최소 `symbol`, `acceptedSeq`를 포함해 계산 결과의 추적성을 확보한다.
- `acceptedSeq`는 계산 자체보다 추적/재생 메타에 가깝다
- **대응:** `PlaceCalculationResult`가 application engine DTO임을 유지하고, `acceptedSeq`는 trace metadata로 취급한다.

### 결정 2-1 (신규): `CancelCalculationResult`도 sealed result로 표현

cancel 역시 결과 의미를 타입으로 드러낸다.

```java
public sealed interface CancelCalculationResult
    permits CancelCalculationResult.Cancelled,
            CancelCalculationResult.Skipped,
            CancelCalculationResult.Rejected {

    record Cancelled(
        Symbol symbol,
        long acceptedSeq,
        List<Order> updatedOrders,
        List<BookOperation> bookOps
    ) implements CancelCalculationResult {}

    record Skipped(
        Symbol symbol,
        long acceptedSeq,
        CancelResultCode reasonCode
    ) implements CancelCalculationResult {}

    record Rejected(
        Symbol symbol,
        long acceptedSeq,
        CancelResultCode reasonCode
    ) implements CancelCalculationResult {}
}
```

구분 원칙:

- `Cancelled`: 실제로 활성 주문이 제거됨
- `Skipped`: 이미 체결 완료 등으로 live 변경이 필요 없음
- `Rejected`: 잘못된 요청 또는 비즈니스 규칙상 거절

시스템 실패만 예외로 전파한다.

`Cancelled.updatedOrders` 범위:

- 현재 MVP에서는 보통 cancelled order 1건만 포함한다
- 장기적으로 관련 상태 변경이 생겨도 place와 동일한 구조를 유지하기 위해 `List<Order>`로 둔다

사유 코드는 자유 문자열 대신 구조화된 코드로 통일한다.

```java
public enum PlaceRejectCode {
    INVALID_ORDER_TYPE,
    INVALID_TIF,
    INVALID_QUANTITY_MODE
}

public enum CancelResultCode {
    ORDER_NOT_FOUND,
    ORDER_ALREADY_FINAL,
    OWNER_MISMATCH,
    SYMBOL_MISMATCH,
    CANCEL_NOT_ALLOWED
}
```

place / cancel의 의미가 다르므로, MVP에서는 공용 enum 하나보다 **분리된 코드 집합**을 우선한다.
일반 GTC LIMIT 주문에서 "지금 당장 체결 안 됨"은 거절이 아니라 정상 resting 흐름이므로,
`PRICE_NOT_MATCHABLE` 같은 코드는 place 공용 거절 코드에서 제외한다.
또한 `PlaceRejectCode`는 검증 계층 전체의 오류 코드가 아니라, **엔진 계산 계층 고유의 거절 코드만 최소 집합으로 유지**한다.

**cancel 결과 경계 기준:**

| 케이스                                                         | 결과 타입      | 기준                           |
|-------------------------------------------------------------|------------|------------------------------|
| 이미 fully filled / 이미 final state이고 idempotent cancel로 취급 가능 | `Skipped`  | 요청 형식은 유효하지만 live 변경할 대상이 없음 |
| 주문 없음                                                       | `Rejected` | cancel 대상 자체가 유효하지 않음        |
| owner mismatch                                              | `Rejected` | 취소 권한 불충족                    |
| symbol mismatch                                             | `Rejected` | 잘못된 symbol routing 또는 요청 오류  |
| 상태상 cancel 불가                                               | `Rejected` | 비즈니스 규칙 위반                   |

### 결정 4 (개정): `Order.applyFrom()` 및 `OrderBook.updateInPlace()` — **제거**

> **기존 결정 취소**: Order 불변화로 인해 mutation 메서드가 불필요해졌다.

| 기존 결정                  | 기존 내용                               | 변경 후                                    |
|------------------------|-------------------------------------|-----------------------------------------|
| 결정 4 `applyFrom()`     | `Order.applyFrom(Order)` 추가         | **제거** — 불변 객체에 mutation 메서드 불필요        |
| 결정 4 `updateInPlace()` | `OrderBook.updateInPlace(Order)` 추가 | **`OrderBook.replaceOrder(Order)`로 대체** |

**`OrderBook.replaceOrder(Order)` 추가:**
```java
// OrderBook.java에 추가
public void replaceOrder(Order updatedOrder) {
    // Deque 구조(FIFO 위치) 불변 유지
    // index만 새 인스턴스로 교체
    if (index.containsKey(updatedOrder.getOrderId())) {
        index.put(updatedOrder.getOrderId(), updatedOrder);
    }
}
```

**`BookOperation.Replace` 매핑:**
```java
// OrderBookProjectionApplier.java
case BookOperation.Replace r -> orderBook.replaceOrder(r.updatedOrder());
// 기존: orderBook.updateInPlace(r.updatedOrder())
```

**`BookOperation` 의미 고정:**

- `Add`: resting order 신규 등록
- `Replace`: 동일 위치(FIFO/price level/side 유지) 상태 갱신
- `Remove`: resting order 제거

구현 기본안:

- `Add(Order order)`
- `Replace(Order updatedOrder)`
- `Remove(OrderId orderId)`

즉, `Add`는 `orderId`만 들고 있지 않고 apply 시점에 바로 등록 가능한 `Order` payload를 포함한다.

추가 규칙:

- `Add`는 resting order 신규 등록에만 사용한다
- MARKET 주문은 resting book에 남지 않으므로 `Add`/`Replace` 대상이 아니다
- `Replace`의 가격 레벨 검증은 resting 가능한 `LIMIT` 주문에만 적용한다

`OrderBook.add(Order)` 불변식:

- `orderId`가 index에 이미 존재하면 안 된다
- resting 가능한 주문만 추가할 수 있다 (`LIMIT` + 활성 상태)
- MARKET 주문은 추가 대상이 아니다
- side/price level이 유효하지 않으면 invariant 예외로 처리한다

**`replaceOrder()` invariant 검증 필수:**

- 기존 order가 index에 존재해야 함
- `orderId` 동일해야 함
- `side` 동일해야 함
- `symbol` 동일해야 함
- 가격 레벨 동일해야 함 (`LIMIT` 기준)
- 제거된 주문 / 없는 주문 / amend 대상 주문에 사용되면 안 됨

검증 실패는 비즈니스 거절이 아니라 **시스템 invariant 예외**로 본다.

권장 예외 타입:

- `OrderBookInvariantViolationException`
- 또는 더 상위 공통 타입으로 `EngineInvariantViolationException`

MVP에서는 한 가지 명칭으로 통일해 로깅/알람/상태 전이 기준을 단순화한다.

### 결정 5 (개정): `Order.copy()` — **제거**

> **기존 결정 취소**: Order 불변화로 인해 참조 공유가 안전해져 copy()가 불필요해졌다.

불변 객체는 참조를 공유해도 상태 오염이 없으므로 `Order.copy()`를 추가하지 않는다.
다만 안전하게 공유 가능한 것은 `Order` 객체뿐이다.
`OrderBookViewFactory`는 level queue / queue / index 등 **호가창 구조 전체를 live와 분리 복사**해야 한다.

원칙:

- `Order` 인스턴스는 공유 가능
- `OrderBookView`의 컬렉션 구조는 반드시 신규 객체여야 함
- engine 계산 중 `OrderBookView`를 변경해도 live `OrderBook`과 구조 공유가 있으면 안 됨
- `OrderBookViewFactory`는 동일 engine-thread 내부에서만 호출되며, 생성 중 concurrent mutation이 없다는 snapshot consistency 전제를 가진다

### 결정 7 (신규): Order 불변화

**변경 내용:**

```java
// 변경 전: mutable 필드 4개
private Quantity remaining;
private OrderStatus status;
private QuoteQty cumQuoteQty;
private Quantity cumBaseQty;

// 변경 후: 모두 final
private final Quantity remaining;
private final OrderStatus status;
private final QuoteQty cumQuoteQty;
private final Quantity cumBaseQty;
```

상태전이 메서드 시그니처 변경:

```java
// 변경 전 (void, in-place mutation)
public void fill(Quantity, Price)
public void fillQuoteMode(Quantity, Price)
public void cancel()
public void activate()

// 변경 후 (새 Order 인스턴스 반환)
public Order fill(Quantity executedQty, Price executedPrice)
public Order fillQuoteMode(Quantity executedQty, Price executedPrice)
public Order cancel()
public Order activate()
```

단일 private 생성자 유지. 각 메서드는 내부적으로 `new Order(...)` 호출.

**성능 리스크 (GC / allocation pressure):**

`fill()`, `activate()`, `cancel()` 등이 새 `Order` 인스턴스를 반환하면
매칭 루프에서 단기 객체(short-lived object)가 대량 생성될 수 있다.
고TPS 구간에서는 young generation GC 빈도가 증가하고, pause time이 tail latency에 영향을 줄 수 있다.

**현재 결정:**

- 설계 안정성, 계산 순수성, Rust 비교 가능성을 위해 **우선은 불변 `Order`를 유지**한다
- 단, 이 결정은 성능 검증을 통과한다는 전제하에 유지된다

**검증 및 fallback 원칙:**

- 부하 테스트에서 allocation rate / GC pause / TPS가 허용 범위를 벗어나면 재검토한다
- 이 경우의 1차 fallback은 도메인 `Order`를 mutable로 되돌리는 것이 아니라,
  **엔진 내부 전용 mutable 계산 상태(`OrderCalcState` 등) 도입**을 우선 검토한다
- 도메인 `Order`를 mutable로 되돌리는 것은 최후의 선택지로 남긴다
- `OrderCalcState`는 엔진 내부 한 호출 범위 전용으로만 사용하며, 외부 반환 / persistence payload / API 응답 DTO로 사용하지 않는다
- 성능 이슈 발생 시 즉시 대응할 수 있도록 `OrderCalcState` 도입 검토는 별도 후속 티켓으로 분리해 관리한다

즉, "불변 `Order` 채택"은 현재 기본안이지만, **GC/TPS 측정 결과가 나쁘면 계산 전용 mutable state로 보완한다**는 조건부 결정이다.

### 결정 8 (신규): OrderBook 내부 구조 변경 — `Deque<OrderId>` + `index<OrderId, Order>`

**핵심 불변식:**

```
Deque<OrderId> → FIFO 위치 마커 역할 (구조 유지용)
Map<OrderId, Order> index → 최신 Order 인스턴스의 정규 소스
```

`Deque`에는 `Order` 객체를 넣지 않는다. stale 객체 참조를 원천 차단하기 위해
FIFO 큐에는 `OrderId`만 저장하고, 실제 상태는 항상 `index`에서 조회한다.
즉, **큐는 순서 전용 / index는 상태 전용**이다.

**`peek()` 변경:**
```java
// 변경 후: Deque의 head OrderId → index에서 현재 상태 반환
public Optional<Order> peek(Side side) {
    Deque<OrderId> level = bestLevelOf(side);
    if (level == null || level.isEmpty()) return Optional.empty();
    OrderId id = level.peekFirst();
    return Optional.ofNullable(index.get(id));
}
```

**`remove()` / `poll()`도 함께 재정의되어야 한다:**

```java
public Optional<Order> poll(Side side) {
    Deque<OrderId> level = bestLevelOf(side);
    if (level == null || level.isEmpty()) return Optional.empty();

    OrderId id = level.pollFirst();
    Order current = index.remove(id);
    removeEmptyLevelIfNeeded(side, level);
    return Optional.ofNullable(current);
}
```

`totalAvailableQty()`, `aggregateDepth()`, `remove()` 역시 반드시 `OrderId -> index` 경유로 현재 remaining을 조회한다.

**취소 성능을 위한 지연 삭제(Lazy Removal) 전략:**

주문 취소 시 큐(`Deque`)의 중간 요소를 즉시 삭제(O(N))하지 않고, `index`에서만 제거하여 O(1) 성능을 보장한다.

**`peek()` 및 `poll()` 변경 (Skip & Drop 원칙):**

- 큐에서 꺼낸 `OrderId`가 `index`에 없는 경우(지연 삭제됨), 예외 없이 해당 ID를 버리고 다음 요소를 탐색한다
- 큐에 있는 `OrderId`가 `index`에 없는 경우는 정상적인 지연 삭제 상태이므로 예외를 발생시키지 않는다
- 시스템 예외는 `index`에 객체가 존재하지만, 그 객체의 상태(가격, 방향, resting 가능 여부 등)가 현재 큐의 위치와 불일치하는 데이터 오염 상황에만 한정한다

`remove(OrderId)` 의미:

- 기본적으로 `index`에서만 제거해 취소를 완료한다
- 큐의 stale `OrderId`는 이후 `peek()` / `poll()` / `aggregateDepth()` 과정에서 skip/drop 된다
- 따라서 "queue에는 남아 있으나 index에는 없음"은 lazy removal의 정상 상태다
- queue와 index의 동시 제거는 `poll()`이 head를 실제 소비할 때만 발생한다
- 이 의미는 `OrderBook.remove(OrderId)`와 `OrderBookProjectionApplier.Remove(OrderId)`에 동일하게 적용되어야 한다

`aggregateDepth()` stale 처리 규칙:

- 큐를 순회하며 `OrderId -> index` 조회
- `index`에 없으면 stale id로 간주하고 skip
- `index`에 있으면 현재 `remaining`만 합산
- stale id 자체는 예외가 아니다

즉, queue-index 관계에서 예외 대상은 "stale id 존재"가 아니라, **lazy removal로 설명되지 않는 활성 데이터 오염**이다.

**리스크:**
- 구현 시 한 곳이라도 `Deque`의 값을 상태로 해석하면 불변식이 깨진다
- **대응:** `Deque<Order>` 대신 `Deque<OrderId>`를 강제해 stale object 참조를 구조적으로 차단한다.

### 결정 9 (신규): MatchingEngine 계산 루프 재설계 — 반환값 기반

Order 불변화로 인해 루프 내 로직이 반환값 기반으로 전환된다.

```java
// 변경 전 (mutable)

orderBook.poll(side);     // OrderBook 직접 mutate

// 변경 후 (immutable + BookOperation 누적)
Order filledMaker = maker.fill(qty, price);  // 새 인스턴스
Order filledTaker = taker.fill(qty, price);  // 새 인스턴스

if (filledMaker.isFullyFilled())

    ops.add(new BookOperation.Replace(filledMaker));
```

> 상태 갱신에만 사용한다. price 변경 / amend / 재배치에는 사용하지 않는다.

### 결정 10 (신규): 다중 인스턴스 전략 — Sticky Routing (MVP)

**권장안: Sticky Routing + DB rebuild on startup/failover**

```
Client → Load Balancer (심볼 해시 기반 고정 라우팅)
           ├─ Instance A: BTCUSDT, ETHUSDT, ...
           └─ Instance B: BNBUSDT, SOLUSDT, ...

인스턴스 장애 시:
  → 심볼을 다른 인스턴스로 재배정
  → OrderBookRebuilder.rebuild(symbol) 호출 (DB open orders 기반)
  → 정상화 후 커맨드 수신 재개
```

**필수 전제: Single Owner per Symbol**

같은 시점에 동일 symbol의 command를 처리하는 active engine owner는 반드시 1개여야 한다.
Sticky routing은 성능 최적화가 아니라 **소유권 보장 메커니즘**과 함께 동작해야 한다.

- failover 중 old owner는 quarantine 또는 shutdown 상태여야 함
- new owner는 rebuild 완료 전까지 command를 처리하면 안 됨
- split-brain 가능성이 있으면 해당 symbol은 `HALTED`로 격리

Redis / Kafka 방식 대비 선택 이유:
- 현재 단일 스레드 per 심볼 구조 = Sticky Routing과 동일 아키텍처
- Phase D의 `OrderBookRebuilder`가 장애 복구 기반이 됨 (추가 인프라 불필요)
- 매칭 루프 내 네트워크 I/O 없음 → 최저 지연 유지

**인터페이스 경계 격리 (확장 준비):**

현재 구현체는 로컬이지만, 인터페이스로 분리해두어 나중에 교체 가능하게 설계:

```java
// 커맨드 수신 추상화 (현재: BlockingQueue, 미래: Kafka/Redis Stream)
public interface OrderCommandGateway {
    void submit(Symbol symbol, EngineCommand command);
}

// OrderBook 상태 변경 추상화 (현재: 로컬 메모리, 미래: Redis Lua)
public interface OrderBookStateApplier {
    void apply(Symbol symbol, List<BookOperation> ops);
}
```

> **MVP 범위 제한**: 인터페이스만 정의. 구현체는 기존 로컬 방식 유지.
> Redis/Kafka 구현체는 별도 티켓으로 분리.

### 결정 11 (신규): 심볼 상태머신 도입

복구/장애 처리 정책을 일관되게 하기 위해 symbol 단위 상태를 명시한다.

```java
public enum EngineSymbolState {
    ACTIVE,
    REBUILDING,
    DIRTY,
    HALTED
}
```

- `ACTIVE`: 정상 처리 가능
- `REBUILDING`: 신규 command 처리 중지. MVP에서는 ingress 차단 또는 reject로 고정
- `DIRTY`: live/DB 정합성 미보장. 자동 처리 금지
- `HALTED`: split-brain 또는 치명적 invariant 위반. 운영 개입 필요

`apply()` 실패, rebuild 진행, failover 진행은 모두 이 상태를 통해 제어한다.

상태의 source of truth는 symbol별 runtime owner가 가진다.
MVP에서는 `EngineRuntime` 또는 그에 준하는 symbol runtime 컨테이너가 상태를 소유하고,
상태 전이는 `EngineHandler` / rebuild 경로 / failover 제어 로직이 이 owner를 통해 수행한다.
즉, 상태는 여러 컴포넌트에 분산 저장하지 않는다.

**MVP ingress 정책:**

- `REBUILDING` 중에는 신규 command를 처리하지 않는다
- 기본 정책은 `OrderCommandGateway` 또는 상위 ingress에서 해당 symbol submit을 reject한다
- backlog 적재는 별도 티켓으로 분리한다

rebuild는 해당 symbol 처리 중단을 전제로 engine-thread에서 수행하며,
MVP에서는 availability보다 **정합성**을 우선한다.

**권장 상태 전이:**

| 현재 상태        | 이벤트                                    | 다음 상태        |
|--------------|----------------------------------------|--------------|
| `ACTIVE`     | `apply()` 실패 / failover 시작             | `REBUILDING` |
| `REBUILDING` | rebuild 성공                             | `ACTIVE`     |
| `REBUILDING` | rebuild 실패                             | `DIRTY`      |
| `DIRTY`      | split-brain 감지 / 치명적 invariant / 수동 격상 | `HALTED`     |
| `HALTED`     | 수동 복구 완료                               | `ACTIVE`     |

기본 승격 기준:

- 자동 복구 실패 또는 persistence-model inconsistency 확인 → `DIRTY`
- split-brain, 치명적 invariant 위반, 잘못된 owner 동시 활성화 → `HALTED`

`DIRTY` 상태 허용 작업:

- 조회(read-only) 작업
- 관리자/운영자 수동 rebuild 요청
- 수동 recover / 상태 점검

`DIRTY` 상태 금지 작업:

- 신규 주문/취소 자동 처리
- 자동 live projection 적용
- 자동 상태 복귀

### 결정 12 (신규): calculate 실패 정책 — 비즈니스 거절 vs 시스템 실패 분리

계산 단계 실패를 일괄 drop 하지 않는다.

- **비즈니스 거절**
  - 예: 취소 대상 없음, 이미 최종 상태, 가격 조건 불충족에 따른 정상 reject
  - 처리: 사용자에게 reject 응답, live `OrderBook` 변경 없음
- **시스템 실패**
  - 예: NPE, index/queue mismatch, invariant 위반, 예상 못한 상태 전이
  - 처리: 해당 symbol을 `DIRTY` 또는 `HALTED`로 전환하고 알람. 임의 drop 금지

즉, "계산 실패 = drop"이 아니라 "정상 거절만 reject, 시스템 실패는 격리"를 원칙으로 한다.

### 결정 13 (신규): `EngineResultPersistenceService`는 orchestration에 집중

이 서비스는 트랜잭션 경계와 호출 순서를 관리하되, 모든 영속 규칙을 한 클래스에 몰아넣지 않는다.

권장 내부 분리:

- `OrderPersistenceWriter`
- `TradePersistenceWriter`
- `BalanceSettlementWriter`

최소한 public 메서드는 책임별로 분리한다.

- `persistPlaceResult(...)`
- `persistCancelResult(...)`
- `persistRecoveryProjection(...)` (미래 대비)

원칙: `EngineResultPersistenceService`는 "무엇을 어떤 트랜잭션으로 저장할지"를 조율하고,
세부 저장 로직의 거대 클래스가 되지 않도록 제한한다.

persist 실패는 최소 두 범주로 분류한다.

- **retryable persistence error**
  - 예: lock timeout, deadlock retryable, transient DB unavailable
  - 처리: live 반영/캐시 갱신 생략
  - MVP 기본 정책: engine-thread 내 즉시 재시도는 하지 않고, command 실패로 처리한 뒤 상위 오류 응답/알람으로 종료
  - API 의미: 기본적으로 retryable 5xx 계열 오류로 취급하고, idempotency key 재사용 시 중복 반영 없이 동일 실패 맥락을 복원할 수 있어야 한다
  - 같은 idempotency key 재요청 시 MVP 기본 정책은 새 처리 시도가 아니라 **이전 실패 결과를 그대로 반환**하는 것이다
- **logical inconsistency / invariant violation**
  - 예: unique/foreign key violation that should never happen, balance settlement invariant 위반,
    engine result와 persistence model 불일치
  - 처리: MVP 기본값은 `DIRTY` 승격. split-brain/owner 충돌/치명적 invariant 동반 시에만 `HALTED`

### 결정 14 (신규): `MatchingEngine` 및 계산 계층은 완전한 POJO로 유지

향후 `MatchingEngine`을 Rust로 재작성해 결과를 비교하거나 대체 구현으로 붙일 수 있도록,

- `domain/service/MatchingEngine`

- `domain/service/dto/*CalculationInput`
- `application/engine/dto/*CalculationResult`
- 계산에 필요한 순수 도메인 모델 (`Order`, `Trade`, value objects)

**금지 사항:**

- Spring annotation 사용 금지 (`@Service`, `@Component`, `@Transactional` 등)
- Spring framework 타입 import 금지

**허용 사항:**

- JDK 표준 라이브러리
- 순수 도메인 모델 / 값 객체
- 계산 결과를 표현하는 POJO / record

**구조 원칙:**

즉, `MatchingEngine`은 "Spring에서 호출되는 클래스"일 수는 있어도,
클래스 자체는 Spring 없이는 컴파일/실행 불가능한 형태가 되어서는 안 된다.

### 결정 15 (신규): `OrderBookProjectionApplier.apply()` 멱등성 규칙

복구/예외 경계에서 동일 projection이 중복 적용될 가능성을 고려해,
`OrderBookProjectionApplier`의 operation 적용 규칙을 명시한다.

- `Add(Order)`:
  - 동일 `orderId`가 이미 book/index에 존재하면 중복 적용으로 간주
  - MVP에서는 invariant 예외로 처리한다
- `Replace(Order)`:
  - 기존 order가 존재하고 invariant를 만족할 때만 적용 가능
  - 동일 상태 재적용은 허용 가능하나, 상태 불일치 재적용은 invariant 예외
  - "동일 상태"의 기준은 `orderId`, `status`, `remaining`, `cumBaseQty`, `cumQuoteQty`가 모두 동일한 경우다
  - 객체 참조 동일성은 요구하지 않는다
- `Remove(OrderId)`:
  - 기본적으로 `index` 제거만 보장한다
  - queue에 stale id가 남는 것은 lazy removal의 정상 상태다
  - 이미 `index`에 없는 경우 중복 remove로 간주하며, MVP 기본 정책은 no-op 허용이다
  - 즉, `OrderBook.remove(OrderId)`와 동일한 lazy removal semantics를 공유해야 한다

즉, MVP에서 applier는 "느슨한 멱등성"보다 **정합성 검증 우선**으로 설계한다.
중복 적용을 정상 플로우로 허용하는 완전 멱등 모델은 별도 티켓으로 분리한다.

3. `OrderBookViewFactory` 도입 — `Order`는 공유 가능하되, level queue / queue / index 구조는 live와 완전 분리 복사
4. `MatchingEngine`에서 `OrderBook` 필드 제거
5. `MatchingEngine.calculatePlace()` — **반환값 기반 루프**로 재구현
6. `MatchingEngine.calculateCancel()` 구현
7. 비즈니스 거절은 result type, 시스템 실패만 예외로 분리
8. `MatchingEngine` / calculation DTO가 Spring 없이 컴파일 가능한지 검증
9. `EngineRuntime` 생성자 변경

### Phase C: 반영 분리 (EngineHandler 재작성)

> **선행 조건:** `EngineHandler` 통합 테스트 작성 완료 후 진입

1. `EngineResultPersistenceService` 도입
    - `persistPlaceResult(PlaceCalculationResult.Accepted)`: orders/trades/balances 트랜잭션
    - `persistCancelResult(CancelCalculationResult.Cancelled)`: order/balance 트랜잭션
    - 내부 writer 분리로 비대화 방지
2. `OrderBookProjectionApplier` 도입
    - `add` → `orderBook.add()`
    - `remove` → `orderBook.remove()`
    - `replace` → `orderBook.replaceOrder()` (구 `updateInPlace` 대체)
3. `OrderBookProjectionApplier.apply()`의 정합성 우선 규칙 문서/코드 반영
4. `apply()` 실패 시 즉시 `REBUILDING` 전이 + rebuild 시도 경로를 Phase C에 포함
   - rebuild는 engine-thread에서 수행하며, MVP에서는 availability보다 정합성을 우선
5. `EngineHandler` 재작성 — 계산 → 영속 → live 반영 → 캐시 순서 엄수
6. 비즈니스 거절은 reject 응답, 시스템 실패는 `DIRTY/HALTED` 전이 + 알람
7. 성공 응답 시점 명시
   - MVP 기본 정책: `persist` 성공 + live `apply` 성공 이후 성공 응답
   - 캐시 갱신은 성공 응답 직후 후행 단계로 동기 시도한다
   - 캐시 갱신 실패는 성공 응답을 뒤집지 않으며, 별도 운영 알람 대상으로 처리한다

### Phase D: 복구 강화 (Sticky Routing 장애 복구 기반)

1. `OrderRepository.findOpenOrdersBySymbol(Symbol)` 추가 (`ORDER BY acceptedSeq ASC` 필수)
2. `OrderBook.from(List<Order>)` 정적 팩토리 추가
   - 입력은 `acceptedSeq ASC`로 정렬된 open orders라는 전제를 가진다
   - 동일 가격 레벨 내 FIFO 재현은 이 정렬 전제에 의존한다
3. `OrderBookRebuilder` 도입
4. rebuild 성공 시 `ACTIVE` 복귀, 실패 시 `DIRTY` 또는 `HALTED` 고착 정책 명시
5. failover 시 old owner quarantine + new owner rebuild 완료 후 수신 재개 정책 명시

## 검증

### 단위 테스트

- `Order.fill()` — 반환된 새 Order에 remaining/status 반영, 원본 Order 불변 확인
- `Order.cancel()` — 반환된 새 Order에 CANCELLED 상태, 원본 불변 확인
- `OrderBook.replaceOrder()` — index 교체 후 peek()가 새 Order 반환, FIFO 위치 유지 확인
- `OrderBook.replaceOrder()` invariant 검증 — side/price/symbol/orderId 불일치 시 시스템 예외 확인
- `OrderBook.add()` invariant 검증 — 중복 `orderId`, MARKET 주문, 잘못된 level 추가 시 예외 확인
- `OrderBook` 내부 불변식 — `Deque<OrderId>` head와 `index` 상태가 항상 정합함을 확인
- `OrderBook.peek()/poll()/remove()` 호출 시 `index`에 없는 만료된 `OrderId`는 예외 없이 정상적으로 무시(skip)되는지 확인
- `OrderBook.peek()/poll()/remove()`는 유효한 객체의 상태 불일치 시에만 invariant 예외가 발생하는지 확인
- `MatchingEngine.calculatePlace()` — live OrderBook mutation 없이 BookOperation 목록 반환 확인
- `MatchingEngine.calculatePlace()` 비즈니스 거절이 예외가 아니라 `Rejected` 결과로 반환되는지 확인
- `MatchingEngine.calculateCancel()`가 `Cancelled/Skipped/Rejected`를 명확히 구분하는지 확인
- `updatedOrders`와 `trades` 순서 규약이 문서 정의와 일치하는지 확인
- `OrderBookProjectionApplier` — add/remove/replace 각각 적용 후 OrderBook 상태 검증
- `OrderBookProjectionApplier` 중복 적용 시 정합성 우선 규칙대로 invariant 예외가 발생하는지 확인
- 시스템 실패 예외 발생 시 symbol 상태가 `DIRTY/HALTED`로 전이되는지 확인

### 통합 테스트 (`EngineHandler` 기준, Phase C 진입 전 필수)

- DB 반영 성공 시만 live OrderBook 변경 및 캐시 갱신
- DB 반영 실패 시 live OrderBook 이전 상태 유지 확인
- DB 반영 실패 시 캐시 갱신 생략 확인
- cancel skip(이미 체결된 주문) 처리
- place 비즈니스 거절 시 live OrderBook / 캐시 변경 없음
- cancel rejected / skipped 시 영속 및 live 반영이 생략되는지 확인
- persist retryable 실패와 정합성 실패가 서로 다른 상태 전이 정책을 따르는지 확인
- place 시스템 실패 시 symbol `DIRTY/HALTED` 전이 및 후속 처리 중단
- `apply()` 실패 → rebuild → 캐시 재동기화 확인
- rebuild 실패 → `DIRTY/HALTED` 고착 확인
- 성공 응답이 `persist + apply` 이후에만 반환되는지 확인
- cache 갱신 실패가 성공 응답을 뒤집지 않고 운영 알람만 발생시키는지 확인
- cache stale 상태가 다음 성공 이벤트 overwrite 또는 rebuild 후 재생성으로 복구되는지 확인
- `DIRTY` 상태에서 조회/수동 rebuild는 가능하고 자동 주문 처리는 금지되는지 확인
